# AI Sync Manager 终端架构详解

## 1. 文档目的

本文档用于解释当前终端化版本的 AI Sync Manager 架构，重点回答两个问题：

- 程序整体是如何从入口一路执行到具体业务的
- 仓库中每个主要代码文件分别负责什么

本文档描述的是当前仓库里的实际实现，而不是抽象计划稿。

## 2. 当前系统结论

当前项目已经收敛为纯终端架构：

- `Cobra` 负责命令行命令树
- `Bubble Tea` 负责终端交互界面
- `runtime` 负责日志、数据库、底层服务初始化
- `usecase` 负责把“扫描、创建快照、列快照”组织成统一业务入口
- `service` 和 `database` 负责底层能力

当前对外入口只有一个：

```text
cmd/fl/main.go
```

当前主要用户路径：

- `fl scan` — 扫描工具配置
- `fl snapshot create / list / delete / restore / diff` — 快照管理
- `fl remote add / list / remove` — 远端仓库配置
- `fl sync push / pull` — 同步配置到远端仓库
- `fl tui` — 终端交互界面

## 3. 执行链路总览

### 3.1 CLI 命令链路

`scan` / `snapshot` / `remote` / `sync` 的主链路如下：

```text
cmd/fl/main.go
  -> internal/app/runtime/runtime.go
  -> internal/app/usecase/local_workflow.go
  -> internal/cli/cobra/*.go
  -> stdout / stderr
```

含义：

1. `main.go` 创建运行时
2. `runtime.go` 初始化日志、数据库、工具检测器、快照服务
3. `local_workflow.go` 提供统一业务接口
4. Cobra 命令文件解析参数、调用 workflow、格式化输出

### 3.2 TUI 链路

`fl tui` 的主链路如下：

```text
cmd/fl/main.go
  -> internal/app/runtime/runtime.go
  -> internal/app/usecase/local_workflow.go
  -> internal/tui/app.go
  -> internal/tui/model.go
  -> internal/tui/update.go
  -> internal/tui/view.go
```

含义：

1. `main.go` 仍然先初始化 runtime 和 workflow
2. Cobra 的 `tui` 子命令只负责启动 TUI runner
3. TUI 内部通过 Bubble Tea 的 `Model / Update / View` 驱动页面切换
4. 真正的业务调用仍然走 workflow，而不是 TUI 直接访问 service 或 database

### 3.3 按命令拆分的调用关系图

如果你不是想理解“分层”，而是想顺着某条用户命令一直看到最终落在哪些文件，可以直接看这一节。

#### 3.3.1 `fl scan`

```text
cmd/fl/main.go
  -> runtime.New()
  -> usecase.NewLocalWorkflow()
  -> cobra.Execute()
  -> internal/cli/cobra/scan.go:newScanCommand()
  -> LocalWorkflow.Scan()
  -> ToolDetector.DetectWithOptions()
  -> ToolDetector.DetectTool()
  -> ToolDetector.scanCodexConfig() / scanClaudeConfig()
  -> local_workflow.go:buildToolSummary()
  -> root.go:printScanResult()
  -> stdout
```

这一条链路里各文件的职责分工是：

- `cmd/fl/main.go` 负责把 runtime、workflow、CLI 依赖装配起来
- `internal/cli/cobra/scan.go` 只负责注册 `scan` 子命令和调用 workflow
- `internal/app/usecase/local_workflow.go` 负责把 detector 返回的原始结果转成用户可读结构
- `internal/service/tool/detector.go` 负责真正扫描 `Codex` / `Claude` 配置目录
- `internal/cli/cobra/root.go` 负责把结果格式化成中文终端输出

所以 `scan` 的核心边界很清楚：

- 是否检测到配置，由 `detector.go` 决定
- 显示成什么中文说明，由 `local_workflow.go` 和 `root.go` 决定

#### 3.3.2 `fl snapshot create`

```text
cmd/fl/main.go
  -> runtime.New()
  -> usecase.NewLocalWorkflow()
  -> cobra.Execute()
  -> internal/cli/cobra/snapshot_create.go:newSnapshotCreateCommand()
  -> LocalWorkflow.CreateSnapshot()
  -> snapshot.Service.CreateSnapshot()
  -> Collector.Collect()
  -> Collector.collectGlobalFiles() / collectProjectFiles()
  -> database.NewSnapshotDAO().Create()
  -> root.go:printCreatedSnapshot()
  -> stdout
```

这一条链路里要注意三个关键文件：

- `internal/cli/cobra/snapshot_create.go`
  负责解析 `--tools`、`--message`、`--name`、`--scope`、`--project-path`
- `internal/app/usecase/local_workflow.go`
  负责校验输入是否为空，并把 service 错误翻译成用户可读错误
- `internal/service/snapshot/service.go`
  负责真正创建快照、生成 ID、构造元数据、写数据库

继续往下看，实际“收集文件”的逻辑不在 `service.go` 本身，而在：

- `internal/service/snapshot/collector.go`
  负责遍历工具配置目录、读取文件内容、计算哈希、组装 `SnapshotFile`

最终落库点在：

- `internal/database/dao.go`
  由 `SnapshotDAO.Create()` 执行快照元数据写入

也就是说，`snapshot create` 可以拆成四段：

1. CLI 解析参数
2. workflow 校验与翻译错误
3. snapshot service 收集并组装快照
4. DAO 写入 SQLite

#### 3.3.3 `fl snapshot list`

```text
cmd/fl/main.go
  -> runtime.New()
  -> usecase.NewLocalWorkflow()
  -> cobra.Execute()
  -> internal/cli/cobra/snapshot_list.go:newSnapshotListCommand()
  -> LocalWorkflow.ListSnapshots()
  -> snapshot.Service.ListSnapshots()
  -> SnapshotDAO.List()
  -> snapshot.Service.CountSnapshots()
  -> SnapshotDAO.Count()
  -> root.go:printSnapshotList()
  -> stdout
```

这一条链路比创建快照短很多，主要是“查询并格式化”：

- `snapshot_list.go` 解析 `--limit` 和 `--offset`
- `local_workflow.go` 负责处理默认分页值，并把数据库错误转换成用户错误
- `snapshot/service.go` 只是转发到 DAO
- `database/dao.go` 负责真正查询 `snapshots` 表
- `root.go` 负责输出快照列表或空状态

所以如果以后你发现“列表为空”“分页不对”“输出格式不好读”，通常对应的修改点分别是：

- 数据没查出来：`dao.go`
- 默认值不合理：`local_workflow.go`
- 展示不好：`root.go`

#### 3.3.4 `fl tui`

```text
cmd/fl/main.go
  -> runtime.New()
  -> usecase.NewLocalWorkflow()
  -> tui.NewRunner()
  -> cobra.Execute()
  -> internal/cli/cobra/tui.go:newTUICommand()
  -> Runner.Run()
  -> Bubble Tea Program.Run()
  -> Model.Update()
  -> updateHomeKey() / updateCreateKey() / updateDetailKey()
  -> loadScanCmd() / loadSnapshotsCmd() / createSnapshotCmd()
  -> workflow.Scan() / workflow.ListSnapshots() / workflow.CreateSnapshot()
  -> view.go / page_scan.go / page_snapshots.go
  -> terminal UI
```

TUI 链路和 CLI 的最大区别是：

- CLI 是“一次命令，一次输出”
- TUI 是“一个长期存活的状态机，不断收消息和重绘界面”

这里最关键的文件分工是：

- `internal/tui/app.go`
  负责创建并启动 Bubble Tea Program
- `internal/tui/model.go`
  负责定义页面、表单、状态字段
- `internal/tui/update.go`
  负责处理按键和业务消息，是 TUI 的事件调度中心
- `internal/tui/form_create.go`
  负责创建快照表单的输入处理，以及提交后串联“创建快照 -> 刷新列表”
- `internal/tui/view.go`
  负责首页和创建页渲染
- `internal/tui/page_scan.go`
  负责扫描结果页渲染
- `internal/tui/page_snapshots.go`
  负责快照列表页渲染

如果按用户动作去理解，TUI 内部又可以拆成三条小链路：

1. 首页进入扫描页

```text
updateHomeKey()
  -> loadScanCmd()
  -> workflow.Scan()
  -> scanLoadedMsg
  -> Model.Update()
  -> Page = PageScan
  -> page_scan.go 渲染
```

2. 首页进入快照列表页

```text
updateHomeKey()
  -> loadSnapshotsCmd()
  -> workflow.ListSnapshots()
  -> snapshotsLoadedMsg
  -> Model.Update()
  -> Page = PageSnapshots
  -> page_snapshots.go 渲染
```

3. 创建快照并自动跳转列表

```text
updateCreateKey()
  -> createSnapshotCmd()
  -> workflow.CreateSnapshot()
  -> workflow.ListSnapshots()
  -> createSnapshotMsg
  -> Model.Update()
  -> Page = PageSnapshots
  -> StatusMessage = "已创建快照 ..."
  -> page_snapshots.go 渲染
```

因此，TUI 的修改定位也比较直接：

- 交互有问题，看 `update.go`
- 表单有问题，看 `form_create.go`
- 页面不好看或信息不全，看 `view.go` / `page_*.go`
- 数据不对，再往下追到 workflow 和 service

## 4. 分层说明

### 4.1 入口层

职责：

- 组装依赖
- 选择 CLI 或 TUI 路径
- 控制进程退出码

关键特点：

- 不承载业务规则
- 不直接扫描文件
- 不直接写数据库

### 4.2 应用层

职责：

- 把底层 detector / snapshot service 组合成统一的业务接口
- 把底层技术性错误转换成用户可读错误
- 决定对外返回的数据结构

关键特点：

- CLI 和 TUI 共用
- 这里定义了“扫描结果该长什么样”“快照结果该长什么样”

### 4.3 交互层

职责：

- Cobra：命令树、参数解析、命令输出
- TUI：终端页面状态、按键事件、视图渲染

关键特点：

- 不直接碰数据库
- 只调用 workflow

### 4.4 核心服务层

职责：

- 工具配置扫描
- 快照采集、比较、应用
- Git 操作
- 同步打包

关键特点：

- 偏“能力模块”
- 由 runtime 注入到 workflow

### 4.5 基础设施层

职责：

- SQLite 初始化
- DAO 查询
- 日志初始化
- 通用工具函数

### 5.8 `pkg/git/`

这一层提供 Git 操作能力，已从 `internal/service/git/` 迁移至 `pkg/`，作为通用基础能力供 usecase 层调用。当前已被 `fl sync push/pull` 和 `fl remote add` 直接使用。

#### `pkg/git/client.go`

Git 客户端主实现。

职责：

- Clone / Pull / Push / Fetch / Commit
- 仓库状态读取（GetStatus / GetRepositoryInfo）
- 分支管理（EnsureBranch / ListBranches）
- 提交历史与文件对比（Log / GetChangedFiles / GetFileContent）
- HEAD 哈希获取（GetHeadHash / GetRemoteHeadHash）

关键设计：

- Push 时使用 `HEAD:refs/heads/branch` 回退 refspec，解决空仓库/新仓库推送问题
- `git.NoErrAlreadyUpToDate` 在 Push/Pull/Fetch 中视为成功
- `InitRepository` 后自动调用 `EnsureBranch` 将 HEAD 指向 `main`

#### `pkg/git/auth.go`

Git 认证逻辑。

职责：

- SSH / Token / Basic 认证配置处理
- 自动从 URL 推导认证类型（`ssh://` → SSH, `https://` → Token）
- SSH 密钥自动发现（`.ssh/` 目录）和 `.pub` → 私钥自动修正
- 使用 `ssh.InsecureIgnoreHostKey()` 避免 Windows knownhosts 校验失败

## 8. 当前架构的优点

当前实现已经具备几个明显优势：

## 9. 当前仍然存在的结构性问题

虽然主体已经终端化，但仓库里仍有一些历史残留或可继续收敛的地方：

- `README.md` 仍残留旧 GUI 描述
- `internal/service/crypto/` 加密服务已实现但尚未接入业务流程（RemoteConfig 敏感字段仍为明文存储）
- `internal/service/sync/` 还没有从 CLI/TUI 主路径接入（同步功能目前由 usecase 层直接编排）
- 部分文档与当前实现可能还未完全同步

这些问题不影响当前终端程序运行，但会影响后续维护成本。

## 10. 总结

当前项目的本质已经不是“桌面应用残留项目”，而是：

> 一个以 `runtime + usecase + Cobra CLI + Bubble Tea TUI` 为主干的纯 Go 终端程序

如果你后续要继续演进这个仓库，最重要的边界不要破坏：

- `main` 只装配，不承载业务
- CLI/TUI 只调 workflow，不直接碰 database
- detector / snapshot / database 继续保持独立职责
- 文档随实现同步更新
