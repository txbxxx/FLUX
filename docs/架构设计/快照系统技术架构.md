# 快照系统技术架构

## 1. 文档目的

本文档面向开发者，详细描述 AI Sync Manager 快照系统的分层架构、数据模型、核心流程和关键实现细节。

## 2. 快照是什么

快照是 AI 工具配置文件在某一时间点的完整备份。它将 Codex、Claude 等 AI 工具的配置文件（config.toml、settings.json、CLAUDE.md、技能目录、命令目录等）一次性收集、打包并持久化到本地 SQLite 数据库。

用户可以在配置变更前创建快照，日后通过快照列表回溯历史状态，或在需要时将快照内容导出/恢复到指定目录。

## 3. 分层架构

快照系统遵循项目的统一分层规范，自上而下分为四层：

```
┌─────────────────────────────────────────────────────┐
│  CLI / TUI 层                                        │
│  internal/cli/cobra/   internal/tui/                 │
│  职责：解析命令参数、收集用户输入、展示结果             │
│  不包含业务逻辑，仅调用 UseCase 层                     │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  UseCase 层                                          │
│  internal/app/usecase/local_workflow.go               │
│  职责：编排"扫描 → 收集 → 创建快照 → 列表"的完整流程  │
│  协调 Service 层完成业务，返回面向用户的结果结构体      │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  Service 层                                          │
│  internal/service/snapshot/                          │
│  ├── service.go    快照 CRUD、导出、备份、校验、恢复  │
│  ├── collector.go  文件收集（核心逻辑）               │
│  ├── applier.go    快照应用（恢复写入）               │
│  └── comparator.go 快照对比（预留）                   │
│  职责：单一业务领域逻辑，可调用 DAO                    │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  DAO 层                                              │
│  internal/models/snapshot.go                         │
│  职责：SQLite 表的增删改查，序列化/反序列化 JSON 字段  │
└─────────────────────────────────────────────────────┘
```

## 4. 数据模型

### 4.1 核心结构体

```go
// Snapshot — 快照主记录
type Snapshot struct {
    ID          string           // UUID，全局唯一
    Name        string           // 用户指定的名称（可选，默认自动生成）
    Message     string           // 快照说明（必填）
    CreatedAt   time.Time        // 创建时间
    Tools       []string         // 包含的工具列表，如 ["codex", "claude"]
    Metadata    SnapshotMetadata // 元数据（OS、版本、范围等）
    Files       []SnapshotFile   // 本次快照收集的所有文件
    CommitHash  string           // Git 提交哈希（推送后填充）
    Tags        []string         // 用户标签
}

// SnapshotFile — 快照中的单个文件
type SnapshotFile struct {
    Path         string       // 相对路径（如 Users/tanchang/.codex/config.toml）
    OriginalPath string       // 创建时的绝对路径
    Size         int64        // 文件大小（字节）
    Hash         string       // SHA-256 内容哈希
    ModifiedAt   time.Time    // 文件修改时间
    Content      []byte       // 文件完整内容（不参与 JSON 序列化）
    ToolType     string       // 所属工具：codex / claude
    Category     FileCategory // 文件类别
    IsBinary     bool         // 是否为二进制文件
    IsSymlink    bool         // 是否来自符号链接目录（如 ~/.claude/skills/lark-*）
    LinkTarget   string       // 符号链接指向的真实路径（仅 IsSymlink=true 时有值）
}

// SnapshotMetadata — 元数据
type SnapshotMetadata struct {
    OSVersion   string            // 操作系统信息，如 "windows/amd64"
    AppVersion  string            // 应用版本
    ProjectPath string            // 项目路径（项目级快照时填充）
    Extra       map[string]string // 扩展字段
}

// SnapshotPackage — 快照包（Service 层返回给 UseCase 的完整结果）
type SnapshotPackage struct {
    Snapshot    *Snapshot  // 主快照
    Size        int64      // 所有文件总大小
    FileCount   int        // 文件数量
    Checksum    string     // 整包校验和
}
```

### 4.2 项目模型

快照不再区分 global/project/both 三种 scope，而是统一基于**项目模型**。

| 项目类型 | 名称示例 | 收集范围 |
|----------|----------|----------|
| 全局项目 | `codex-global`、`claude-global` | `~/.codex`、`~/.claude` 等用户目录下的配置 |
| 用户注册项目 | 自定义项目名 | 通过 `scan add --project` 注册的项目目录下的配置 |

创建快照时通过 `--project` 指定项目名称，系统根据项目类型自动确定收集范围。

### 4.3 文件类别

| Category | 说明 | 典型文件 |
|----------|------|----------|
| `config` | 配置文件 | config.toml、settings.json |
| `skills` | 技能文件 | skills.yml |
| `commands` | 命令文件 | commands.yml |
| `agents` | Agent 文件 | AGENTS.md |
| `rules` | 规则文件 | rules.md |
| `plugins` | 插件文件 | plugins.yml |
| `mcp` | MCP 配置 | 含 mcp 的路径 |
| `docs` | 文档文件 | *.md |
| `output` | 输出样式 | output-styles 目录 |
| `other` | 其他 | 未匹配的文件 |

### 4.4 数据库表结构

快照主表 `snapshots`：

```sql
CREATE TABLE snapshots (
    id          TEXT PRIMARY KEY,
    name        TEXT,
    description TEXT,
    message     TEXT,
    created_at  INTEGER,          -- Unix 时间戳
    tools       TEXT,             -- JSON 数组，如 ["codex","claude"]
    metadata    TEXT,             -- JSON 对象
    tags        TEXT,             -- JSON 数组
    commit_hash TEXT,
    file_count  INTEGER,
    total_size  INTEGER
);
```

`tools`、`metadata`、`tags` 字段以 JSON 文本存储，DAO 层负责序列化/反序列化。

## 5. 核心流程

### 5.1 创建快照

```
用户输入                          系统处理
────────  ─────────────────────────────────────────────
指定工具    →  UseCase 解析工具列表，校验必填参数
指定说明    →  UseCase 校验 message 非空
           →  Service.CreateSnapshot()
              │
              ├─ Collector.Collect()
              │  │
              │  ├─ RuleResolver 解析扫描规则
              │  │   得到全局规则命中 + 自定义规则命中
              │  │
              │  ├─ 遍历每个命中的目录/文件
              │  │   ├─ 如果是目录 → filepath.Walk 递归收集
              │  │   ├─ 如果是文件 → 直接读取
              │  │   │
              │  │   └─ 对每个文件：
              │  │       ├─ 读取内容 (os.ReadFile)
              │  │       ├─ 计算 SHA-256 哈希
              │  │       ├─ 检测是否二进制（前 512 字节）
              │  │       ├─ 自动分类 (categorizeFile)
              │  │       ├─ seen 去重
              │  │       └─ 构建 SnapshotFile
              │  │
              │  └─ 返回 CollectResult { Files, TotalSize }
              │
              ├─ 生成 UUID 作为快照 ID
              ├─ 构建 Snapshot 和 SnapshotMetadata
              ├─ SnapshotDAO.Create() → 写入 SQLite
              └─ 返回 SnapshotPackage { FileCount, Size, Checksum }
```

### 5.2 文件收集细节

`Collector` 的收集策略不是"打包整个目录"，而是**基于统一扫描规则精确命中**：

1. **规则解析**：`RuleResolver` 根据内置默认规则 + 用户自定义规则 + 已注册项目规则，确定要扫描的路径列表
2. **逐条匹配**：对每条规则命中的路径，判断是目录还是文件
3. **目录递归**：如果是目录，`filepath.Walk` 遍历其下所有文件（受排除规则约束）
4. **文件读取**：直接读取文件内容，计算哈希，自动分类
5. **去重机制**：`seen map[string]struct{}` 确保同一绝对路径只被收集一次

### 5.3 列出快照

```
CLI/TUI                     系统
──────  ─────────────────────────────
指定参数  →  UseCase.ListSnapshots(limit, offset)
           →  Service.ListSnapshots()
              →  SnapshotDAO.List(limit, offset)
                 →  SELECT ... ORDER BY created_at DESC
                 →  反序列化 JSON 字段
              →  返回 []*Snapshot
```

### 5.4 导出快照

```
指定快照 ID + 目标目录
    →  Service.GetSnapshot(id)
    →  遍历 snapshot.Files
       →  拼接 exportPath + file.Path
       →  CloneFileWithContent(目标路径, file.Content)
    →  恢复原始目录结构
```

## 6. 依赖关系

```
service.go
  ├── 依赖 collector.go (文件收集)
  ├── 依赖 models.SnapshotDAO (数据库操作)
  ├── 依赖 models.CalculateChecksum (校验和计算)
  └── 依赖 tool.RuleResolver (扫描规则解析)

collector.go
  ├── 依赖 tool.RuleResolver (规则解析)
  └── 依赖 os/filepath (文件系统操作)

local_workflow.go (UseCase)
  ├── 依赖 snapshot.Service (通过 SnapshotManager 接口)
  └── 依赖 tool.Detector (通过 Detector 接口)
```
