## 1. 设计背景

快照系统已完成本地能力（创建/列表/删除/恢复/对比），本架构为同步引擎提供技术方案，实现快照更新、远端仓库管理、多设备同步和历史版本管理。

### 核心理念

- **SQLite 是唯一真相源** — 存储最新快照数据
- **Git 对用户完全透明** — 用户不直接接触 Git 命令
- **本地优先** — 没有远端仓库也能使用快照更新功能
- **Push 失败安全** — 网络失败不影响本地 SQLite 数据
- **冲突不自动覆盖** — 检测到冲突必须用户确认

## 2. 架构总览

### 2.1 双重存储模型

```
┌─────────────────────────────────────────────┐
│              SQLite（真相源）                 │
│  snapshots + snapshot_files                  │
│  存储最新快照的完整文件内容                     │
└───────────────┬─────────────────────────────┘
                ↕ 同步（Push/Pull 时读写）
┌───────────────┴─────────────────────────────┐
│       Git 工作目录（版本引擎）                 │
│  ~/.ai-sync-manager/repos/<project>/         │
│  文件内容与 SQLite 镜像，Git 负责版本历史       │
└───────────────┬─────────────────────────────┘
                ↕ push/pull
┌───────────────┴─────────────────────────────┐
│       远端仓库（同步通道）                     │
│  GitHub / Gitea / 自建 Git                   │
└─────────────────────────────────────────────┘
```

### 2.2 数据流方向

- **Push**：SQLite → 工作目录（写文件）→ Git commit → Git push → 远端
- **Pull**：远端 → Git fetch → 对比变更 → 冲突检测 → 更新 SQLite
- **Update**：磁盘文件 → Collector 收集 → 对比哈希 → 更新 SQLite

### 2.3 多 Project 共享仓库

```
repos/
└── shared-repo/              # 本地 Git 仓库
    ├── claude-global/        # project-a 的文件
    │   ├── settings.json
    │   └── CLAUDE.md
    ├── codex-global/         # project-b 的文件
    │   └── config.toml
    └── ...
```

每个 project 按子目录隔离，共享同一个 Git 仓库和远端配置。

## 3. 分层调用链

```
CLI (remote.go / sync.go / snapshot_update.go / snapshot_history.go)
  → UseCase (LocalWorkflow)
    → Service 层
      ├→ SnapshotService   快照 CRUD
      ├→ GitClient         Git 操作
      └→ RemoteConfigDAO   远端配置
        → DAO (SnapshotDAO / RemoteConfigDAO)
```

**绝对禁止**：CLI 直接调用 Service 或 DAO。

## 5. 核心流程

### 5.1 SyncPush 流程

```
1. 参数校验（project 必填，或 --all）
2. 获取活跃远端配置
3. 查找 project 最新快照
4. 确保工作目录存在（clone 或 init）
5. 从 SQLite 读文件 → 写入工作目录
6. Git add + commit（空工作树视为成功）
7. Git push 到远端
```

### 5.2 SyncPull 流程（含冲突检测）

```
1. 参数校验 + 获取远端配置
2. 确保本地仓库存在
3. 记录本地 HEAD → Git fetch
4. 对比 local HEAD 和 remote HEAD
5. 逐文件三方对比：
   - 本地快照 vs 本地 Git HEAD → 判断本地是否修改
   - 本地 Git HEAD vs 远端 HEAD → 判断远端是否修改
   - 双方都修改 → 冲突
6. 有冲突 → 返回冲突列表（不自动合并）
7. 无冲突 → Git pull + 更新 SQLite
```

**设计决策：不做 git merge，改用 fetch + 对比**
- go-git 的 merge 能力有限
- 配置文件冲突需要用户明确选择
- fetch + compare 更安全

### 5.3 SnapshotHistory 流程

```
1. 解析快照 ID/名称
2. 定位 Git 仓库路径（repos/<project>/）
3. 调用 GitClient.Log() 获取 commit 历史
4. 返回 HistoryResult（包含 commit hash、消息、时间）
```

### 5.4 RestoreFromHistory 流程

```
1. 解析快照 ID/名称 + commit hash
2. 从指定 commit 读取所有文件内容
3. 支持 --dry-run 预览模式
4. 恢复文件到原始路径
```

## 6. 远端管理

### 6.1 认证方式

| 类型 | 字段 | 适用场景 |
|------|------|---------|
| `none` | — | 公开仓库 |
| `token` | Token | GitHub/Gitea HTTPS |
| `basic` | Username + Password | 自建 Git HTTPS |
| `ssh` | SSHKey + Passphrase | SSH 协议 |

### 6.2 连通性测试

`remote add` 时会自动测试远端连接：
- 使用浅克隆（depth=1）到临时目录
- 成功 → 状态标记为 `active`
- 失败 → 状态标记为 `error`，仍然保存配置

### 6.3 RemoteConfigDAOAdapter

```go
type RemoteConfigDAOAdapter struct {
    dao *models.RemoteConfigDAO
}
```

适配器模式，将 DAO 方法名映射到 `RemoteConfigManager` 接口。

## 7. 错误处理

| 场景 | 处理方式 | UserError Message |
|------|---------|-------------------|
| 快照不存在 | 报错退出 | "项目 xxx 没有快照" |
| 未配置远端仓库 | 引导操作 | "请先执行 ai-sync remote add <url>" |
| 远端不可达 | 报错展示原因 | "无法推送到远端，请检查网络" |
| Push 失败 | 报错，本地不变 | "本地数据未受影响" |
| Pull 有冲突 | 返回冲突列表 | "检测到 N 个文件冲突" |
| 空工作树 commit | 返回成功 | FilesPushed: 0 |
| 认证失败 | 提示检查凭证 | "请检查 Token/SSH 密钥配置" |
