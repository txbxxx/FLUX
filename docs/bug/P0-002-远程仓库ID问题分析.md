### 进一步调查（2026-04-13）

**实际验证结果**：

执行 `ai-sync scan list` 显示 `claude-global` 项目**存在于数据库中**，且有 5 个可同步项。

**关键发现 - 根本原因确认**：

通过添加调试日志，追踪到错误发生在数据库保存阶段：

```
"保存快照失败","error":"SQL logic error: too many SQL variables (1)"
```

**根本原因**：

SQLite 数据库有 SQL 变量数量限制（通常为 999 或 32766 个变量）。当尝试保存 7979 个文件时，SQL 变量数量远远超过了这个限制，导致插入失败。

具体来说：
- 用户 `~/.claude` 目录下有大量文件（通过符号链接引用了其他目录）
- `collector.Collect` 收集了 7979 个文件
- 但 SQLite 的 `INSERT` 语句中包含 7979 个文件的路径、内容、哈希等信息
- 每个文件需要多个 SQL 变量（路径、原始路径、工具类型、哈希、内容、是否为二进制等）
- 总变量数远远超过 SQLite 的限制

**解决方案**：

1. **分批插入**：将文件列表分成多批插入，每批不超过 SQLite 的变量限制
2. **使用事务**：将大插入分成多个小插入，每个事务处理一部分文件
3. **内容外部存储**：将文件内容存储在外部文件或对象存储中，数据库只存储引用

## 修复方案

### 方案 1: 改进错误信息的传播（推荐）

在 `local_workflow.go` 的 CreateSnapshot 方法中添加更精确的错误识别：

```go
pkg, err := w.snapshots.CreateSnapshot(typesSnapshot.CreateSnapshotOptions{...})
if err != nil {
    errMsg := err.Error()

    // 更精确地识别错误类型并给出针对性建议
    switch {
    case strings.Contains(errMsg, "未找到项目"):
        // 直接返回原始错误，因为已经包含建议
        return nil, &UserError{
            Message:    "创建快照失败：" + errMsg,
            Suggestion: "请先运行 'ai-sync scan add claude' 或 'ai-sync scan add codex' 注册全局项目",
            Err:        err,
        }
    case strings.Contains(errMsg, "未找到任何配置文件"):
        return nil, &UserError{
            Message:    "创建快照失败：未找到任何配置文件",
            Suggestion: "请确认所选工具的配置目录中存在可备份文件",
            Err:        err,
        }
    default:
        return nil, &UserError{
            Message:    "创建快照失败",
            Suggestion: "请检查工具选择和本地配置目录后重试",
            Err:        err,
        }
    }
}
```
