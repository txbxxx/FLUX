# GORM 数据库迁移设计

## 文档信息

- 日期：2026-04-03
- 主题：将项目数据库访问从 `database/sql` 全量迁移到 GORM
- 状态：设计已确认
- 约束：
  - 全量切换，仓库业务数据库访问不再直接依赖 `database/sql`
  - 保持现有 SQLite 数据文件与表结构兼容
  - 旧的 `ai-sync-manager.db` 必须可直接继续使用

## 1. 目标

本次迁移的目标不是“重做数据库层”，而是：

1. 将数据库连接、事务、CRUD 查询从 `database/sql` 切换到 GORM
2. 保持现有 SQLite 文件、表名、列名、索引、触发器行为兼容
3. 尽量不改变上层 `service` / `usecase` / CLI / TUI 的调用方式
4. 为后续继续扩展 DAO 和关联查询提供统一 ORM 基础

本次迁移完成后，项目中的数据库访问风格应收敛为：

- `pkg/database` 负责 GORM 初始化与 SQLite 兼容性保证
- `internal/models/*DAO` 统一通过 GORM 访问数据库
- 少量 SQLite 特定 DDL 通过 `gorm.DB.Exec(...)` 执行

## 2. 当前现状

当前原生 SQL 主要集中在两层。

### 2.1 `pkg/database/db.go`

职责：

- 打开 SQLite 连接
- 设置连接池
- 执行 `PRAGMA foreign_keys = ON`
- 通过手写 SQL 创建表、索引、触发器
- 提供事务辅助、测试库初始化、数据库备份与 `VACUUM`

当前实现直接持有 `*sql.DB` 与 `*sql.Tx`。

### 2.2 `internal/models/*DAO`

当前至少包含以下 DAO：

- `SnapshotDAO`
- `RemoteConfigDAO`
- `SyncTaskDAO`
- `CustomSyncRuleDAO`
- `RegisteredProjectDAO`

这些 DAO 目前都直接使用：

- `Exec`
- `Query`
- `QueryRow`
- `Scan`

并手动处理：

- `sql.ErrNoRows`
- `sql.Null*`
- JSON 字段序列化与反序列化
- Unix 时间戳与 `time.Time` 之间的转换

## 3. 设计原则

### 3.1 业务 CRUD 全部改为 GORM

所有 DAO 的持久化行为统一改为 GORM，不再直接写 `database/sql` CRUD。

允许保留的 SQL 只有：

- SQLite 特定 `PRAGMA`
- `CREATE INDEX IF NOT EXISTS ...`
- `CREATE TRIGGER IF NOT EXISTS ...`
- 少量无法用 Migrator 稳定表达的兼容性 DDL

也就是说，SQL 文本可以保留少量，但执行入口统一变为 GORM。

### 3.2 兼容旧库优先于“纯自动迁移”

本次最重要的约束不是“尽量少写 SQL”，而是：

> 已存在的 `ai-sync-manager.db` 文件必须可以继续直接打开并正常使用。

因此设计以“现有 schema 为准”，不是以 GORM 默认推断结果为准。

### 3.3 分离“访问方式变化”和“物理结构变化”

本次迁移只改变：

- 连接管理方式
- 事务管理方式
- DAO 实现方式

尽量不改变：

- 表名
- 列名
- 索引名
- 触发器行为
- 时间字段的存储语义
- JSON 文本字段的持久化格式

### 3.4 上层接口尽量稳定

`service` 层和 `usecase` 层已经依赖这些 DAO 的行为。

因此本次迁移尽量保持：

- DAO 构造器不大改
- DAO 方法名不大改
- 返回错误语义尽量不变

## 4. 方案选择

### 方案 A：全面切换到 GORM，但保留少量 SQLite DDL SQL

特点：

- `pkg/database` 改为持有 `*gorm.DB`
- DAO CRUD 全部使用 GORM
- 表、索引、触发器兼容性通过 GORM + `Exec` 组合保证

优点：

- 满足“全量切换到 GORM”的目标
- 兼容已有 SQLite 文件风险最低
- 对现有上层代码冲击较小

缺点：

- 不是“零 SQL 字符串”
- 迁移层仍需要维护少量 SQLite DDL

### 方案 B：完全依赖 AutoMigrate / Migrator

特点：

- 试图用 GORM tag 和 AutoMigrate 生成全部结构

优点：

- 表面上最统一

缺点：

- 现有触发器无法稳定表达
- SQLite 索引名、默认值、触发器与时间字段兼容风险高
- 旧库兼容性难以保证

### 方案 C：新增持久化模型层，与当前领域模型完全拆开

特点：

- 领域模型和 ORM 模型分层
- DAO 负责转换

优点：

- 长期维护性最好

缺点：

- 本次改动面最大
- 与“尽快完成主路径迁移”的目标不匹配

### 结论

本次采用：

> 方案 A：业务 CRUD 全量切换到 GORM，兼容性 DDL 通过 GORM 执行少量 SQL。

## 5. 数据库层设计

### 5.1 `pkg/database.DB`

`pkg/database.DB` 改为持有：

- `conn *gorm.DB`
- `path string`
- `closed bool`

保留对外职责：

- 初始化数据库
- 关闭数据库
- 提供事务能力
- 返回数据库路径和文件大小
- 备份数据库
- `VACUUM`

### 5.2 初始化流程

新的初始化流程：

1. 确保数据目录存在
2. 使用 GORM 打开 SQLite 文件
3. 获取底层 `sql.DB` 连接池并设置：
   - `SetMaxOpenConns(1)`
   - `SetMaxIdleConns(1)`
   - `SetConnMaxLifetime(time.Hour)`
4. 执行 `PRAGMA foreign_keys = ON`
5. 执行结构兼容迁移
6. 执行索引与触发器补齐

### 5.3 为什么不能只依赖 AutoMigrate

原因不是 GORM 不能建表，而是：

1. 现有项目有明确的 SQLite trigger
2. 现有项目已经手工定义索引名
3. 时间字段与 JSON 字段的物理存储格式必须兼容旧库
4. 旧库继续使用时，schema 漂移风险必须尽量降到最低

因此本次设计为：

- 基础结构可以使用 GORM migrator 能力
- trigger、部分索引和 SQLite 特定行为仍通过 `gorm.DB.Exec(...)` 固定下来

## 6. 模型兼容策略

### 6.1 时间字段

当前项目大量时间字段最终落成 Unix 秒整数。

本次不直接改成 GORM 默认 datetime 语义，而是保持现有数据库列兼容。

实现方式：

- 领域模型继续暴露 `time.Time` / `*time.Time`
- 持久化层通过自定义 GORM model、hook 或 serializer 保持整数列读写

目标是：

- 老数据可继续读取
- 新数据写回后格式不变

### 6.2 JSON 字段

当前以下字段本质上是 JSON 文本：

- `tools`
- `tags`
- `metadata`
- 其他 DAO 中的复合字段

本次继续保持数据库层为文本列，避免更换物理存储格式。

GORM 层负责：

- 写入前序列化
- 读取后反序列化

### 6.3 主键与表名

保持现有：

- 字符串主键继续是字符串主键
- 表名继续与当前 schema 一致
- 不引入 `gorm.Model`
- 不默认引入软删除列

### 6.4 关联字段

像 `Snapshot` 与 `SnapshotFile` 这种关系，需要补足 GORM 层所需外键映射。

但最终数据库列结构必须与现有表兼容，不能因为 ORM 方便而改动已有表语义。

## 7. DAO 设计

### 7.1 DAO 总体策略

每个 DAO 继续保留当前位置与职责，但内部实现改为 GORM。

统一替换为：

- `Create`
- `Where`
- `First`
- `Find`
- `Delete`
- `Updates`
- `Count`

并统一处理：

- `gorm.ErrRecordNotFound`
- 关联预加载
- JSON 字段序列化
- 时间字段转换

### 7.2 SnapshotDAO

重点：

- 主记录仍落到 `snapshots`
- 文件记录落到 `snapshot_files`
- `Tools` / `Tags` / `Metadata` 继续保持兼容存储格式
- `GetByID` 需要同时恢复主记录和文件集合

### 7.3 RemoteConfigDAO

重点：

- 默认配置查询
- 更新 `is_default`
- 认证相关字段兼容

### 7.4 SyncTaskDAO

重点：

- `started_at` / `completed_at` 空值语义
- 进度字段持久化
- 列表查询与排序

### 7.5 CustomSyncRuleDAO / RegisteredProjectDAO

这两个 DAO 结构最简单，适合作为首批迁移对象，用来先验证数据库底座是否稳定。

## 8. 事务与错误语义

### 8.1 事务

原有：

- `Transaction(func(*sql.Tx) error) error`

改为：

- `Transaction(func(*gorm.DB) error) error`

由数据库层统一包裹，避免调用方自己管理底层事务对象。

### 8.2 错误语义

对上层尽量保持一致。

例如：

- 记录不存在时，DAO 仍返回当前中文错误文本
- 上层 `service` 和 `usecase` 的错误翻译尽量不变

这样可以减少 CLI/TUI 与测试变更面。

## 9. 测试策略

迁移按 TDD 推进，但本次测试重点不是“重建全量覆盖”，而是优先锁住兼容风险最大的路径。

优先级如下：

1. `pkg/database`
2. 简单 DAO：`CustomSyncRuleDAO`、`RegisteredProjectDAO`
3. 中等复杂 DAO：`RemoteConfigDAO`、`SyncTaskDAO`
4. 复杂 DAO：`SnapshotDAO`
5. 依赖数据库的服务测试：
   - `internal/service/tool/...`
   - `internal/service/snapshot/...`
   - `internal/app/runtime/...`

测试要求：

- 测试数据库初始化路径与生产初始化路径保持一致
- 不允许出现“测试库结构和真实库结构不一致”
- 对旧 schema 的兼容以现有表结构为准

## 10. 风险与应对

### 风险 1：GORM 默认行为导致 schema 漂移

应对：

- 以现有建表 SQL 为基准核对 GORM model/tag
- 关键索引和 trigger 直接固定为 `Exec`

### 风险 2：时间字段兼容失败

应对：

- 单独验证旧数据读取和新数据回写
- 不贸然切换到默认 datetime

### 风险 3：DAO 改造引发上层连锁变更

应对：

- 保持 DAO 接口基本稳定
- 先迁简单 DAO，再迁复杂 DAO

### 风险 4：测试环境与真实环境分叉

应对：

- `InitTestDB` 与 `InitDB` 共用同一套 GORM 初始化与迁移逻辑

## 11. 实施结论

本次迁移按照以下总路径推进：

1. 先把数据库底座改成 GORM
2. 再逐个 DAO 迁移
3. 最后跑主路径测试并收尾

最终验收标准：

- 仓库业务数据库访问不再直接依赖 `database/sql`
- SQLite 旧库可直接继续使用
- 主路径测试通过
- CLI / TUI / service 层行为不因 ORM 替换而改变
