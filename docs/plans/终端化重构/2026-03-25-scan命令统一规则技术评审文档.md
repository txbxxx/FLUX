## 1. 评审结论

本次需求不应按“给 `scan` 命令补几个参数”处理，而应按“同步规则统一化”处理。

评审结论如下：

1. 必须引入统一规则源，作为 `scan`、`get`、`snapshot create` 的共同依据。
2. `scan add/remove/list` 不能只改 CLI 层，必须有持久化层和规则解析层。
3. Claude `plugins/` 不能再按整目录同步，必须改为文件级白名单。
4. “可同步 / 暂不可同步 / 不可同步”必须增加关键文件门槛。
5. `snapshot create` 不能继续递归打包整个配置根目录，否则会与 `scan` 展示失真。
6. `get` 的访问白名单必须跟随最终生效规则，否则会出现“scan 看得到但 get 读不到”的断链。

最终建议采用“统一规则源”方案，不建议做局部修补。

## 2. 产品边界复核

### 2.1 本次要做

- 支持 `ai-sync scan`
- 支持 `ai-sync scan codex`
- 支持 `ai-sync scan claude`
- 支持 `ai-sync scan codex claude`
- 支持 `scan add <app> <relative-file-path>`
- 支持 `scan remove <app> <relative-file-path>`
- 支持 `scan list`
- 支持 `scan list <app>`
- 支持关键文件门槛判定
- 支持默认规则与用户自定义规则合并后的最终规则
- 统一 `scan`、`get`、`snapshot create`
- 更新 CLI/TUI/文档展示

### 2.2 本次不做

- 项目级自定义规则
- 自定义目录级规则
- 远端同步联动
- GUI 规则配置
- 动态从官方文档拉规则

## 3. 现状问题复核

结合现有代码，本次需求涉及的问题已经确认如下：

### 3.1 `scan` 入口仍是无参数命令

- `internal/cli/cobra/scan.go` 当前只有无参数 `scan`
- `internal/app/usecase/local_workflow.go` 当前固定返回 `codex` 与 `claude`

影响：

- 无法按 app 过滤
- 无法承载 `scan add/remove/list`

### 3.2 detector 仍以“目录存在 + 命中任意项”判状态

- `internal/service/tool/detector.go` 当前没有关键文件门槛

影响：

- Codex 没有 `~/.codex/config.toml` 时仍可能被判定为可同步
- Claude 没有 `~/.claude/settings.json` 时仍可能被判定为可同步

### 3.3 Claude `plugins/` 仍是整目录规则

- `internal/service/tool/paths.go` 当前把 `plugins/` 作为目录级默认规则

影响：

- 与本次文件级白名单需求冲突
- 与“最小可恢复状态”原则冲突

### 3.4 `get` 与 `snapshot create` 没有共用规则源

- `internal/service/tool/accessor.go` 用静态定义做白名单
- `internal/service/snapshot/collector.go` 直接从根目录递归打包

影响：

- `scan` 展示与 `snapshot create` 打包范围可能不一致
- `scan add` 后如果不联动 accessor，`get` 会读不到新增规则

### 3.5 TUI 展示仍停留在旧模型

- `internal/tui/page_scan.go` 现在只显示工具名、内部状态值、数量

影响：

- 不显示中文状态
- 不显示原因
- 不显示自定义规则命中情况

## 4. 核心产品决策

## 4.1 统一规则源

本次以“统一规则源”作为主线。

统一规则源负责：

- 提供默认规则
- 提供关键文件定义
- 读取用户自定义规则
- 合并得到最终生效规则
- 计算当前命中结果
- 生成状态、原因、提醒信息

调用关系应改为：

- `scan` 负责展示规则解析结果
- `get` 负责校验路径是否属于最终允许访问范围
- `snapshot create` 负责只打包最终命中的文件

## 4.2 Claude `plugins/` 只同步三个文件

这里需要明确产品依据，避免后续误解。

本项目默认只同步：

- `plugins/blocklist.json`
- `plugins/installed_plugins.json`
- `plugins/known_marketplaces.json`

原因不是“官方明确要求只同步这三个文件”，而是：

1. 这三个文件足以表达插件安装状态、市场配置状态和屏蔽状态。
2. 其他插件相关文件属于迁移后可再生成内容，不属于必须持久迁移的源数据。
3. 因此这三个文件满足“最小可恢复状态”原则，适合作为默认同步白名单。

如果用户确有需要，可以通过 `scan add` 显式追加其他插件文件。

## 4.3 关键文件门槛

全局作用域下，只有识别到关键文件，工具才允许进入“可同步”状态。

- Codex 关键文件：`~/.codex/config.toml`
- Claude 关键文件：`~/.claude/settings.json`

这条规则优先于“是否命中其他配置项”。

## 4.4 自定义规则持久化位置

自定义规则必须持久化到 AI Sync Manager 自己的数据目录，不写回用户工具目录。

推荐落在现有 SQLite 数据库中，而不是散落 JSON 文件，原因如下：

1. 当前 runtime 已统一初始化 SQLite。
2. 后续 `scan add/remove/list` 都需要稳定增删查。
3. 数据库天然适合做唯一约束、去重和后续扩展。
4. 能保持与现有应用数据一致的生命周期和备份方式。

## 5. 推荐架构方案

## 5.1 模块拆分

建议在 `internal/service/tool` 下新增统一规则模块，至少拆成以下职责：

### `RuleDefinition`

职责：

- 维护默认规则
- 维护关键文件定义
- 维护标签、分组、是否目录、是否关键文件等元数据

建议文件：

- `internal/service/tool/rule_definitions.go`

### `RuleStore`

职责：

- 读取用户自定义全局文件规则
- 新增规则
- 删除规则
- 列出规则

建议文件：

- `internal/models/custom_sync_rule.go`
- `internal/service/tool/rule_store.go`

### `RuleResolver`

职责：

- 合并默认规则和自定义规则
- 校验路径是否存在
- 判断关键文件状态
- 计算默认命中项、自定义命中项、未命中自定义项
- 产出状态、原因、提醒、最终允许访问路径、最终可打包文件列表

建议文件：

- `internal/service/tool/rule_resolver.go`
- `internal/service/tool/rule_resolver_test.go`

## 5.2 目标调用链

```text
Cobra scan / get / snapshot create
  -> usecase
    -> RuleResolver
      -> RuleDefinition + RuleStore
        -> 文件系统 / SQLite
```

其中：

- detector 不再单独决定最终同步规则
- accessor 不再直接依赖静态 definitions
- snapshot collector 不再直接递归整个根目录

## 6. 规则模型设计

## 6.1 默认规则

### Codex 默认全局规则

- `config.toml`
- `skills/`
- `rules/`
- `superpowers/`

### Codex 默认项目规则

- `.codex/config.toml`
- `AGENTS.md`

### Claude 默认全局规则

- `settings.json`
- `CLAUDE.md`
- `commands/`
- `skills/`
- `output-styles/`
- `plugins/blocklist.json`
- `plugins/installed_plugins.json`
- `plugins/known_marketplaces.json`

### Claude 默认项目规则

- `.claude/`
- `CLAUDE.md`

## 6.2 自定义规则

第一阶段仅支持：

- `scope = global`
- `kind = file`

建议数据字段：

- `id`
- `tool_type`
- `scope`
- `relative_path`
- `created_at`
- `updated_at`

建议唯一约束：

- `(tool_type, scope, relative_path)` 唯一

## 6.3 命中结果模型

建议扫描结果不要只返回扁平 `ConfigFiles`，而要返回可直接供 CLI/TUI/usecase 消费的报告模型。

建议包含：

- 工具名
- 配置根目录
- 中文状态
- 内部状态值
- 原因
- 可同步项数量
- 关键文件列表
- 默认规则命中项
- 自定义规则命中项
- 未命中自定义项
- 提醒信息
- 最终允许访问路径列表
- 最终允许打包文件列表

## 7. 状态判定设计

状态优先级建议固定如下：

### 7.1 不可同步

条件：

- 未找到工具全局配置目录

原因文案：

- `未找到配置目录 <path>`

### 7.2 暂不可同步

条件一：

- 找到配置目录，但缺少关键文件

原因文案示例：

- `缺少关键文件 ~/.codex/config.toml`
- `缺少关键文件 ~/.claude/settings.json`

条件二：

- 找到配置目录
- 找到关键文件
- 但当前最终规则下没有任何命中项

原因文案：

- `找到了配置目录和关键文件，但当前规则下没有识别到可同步项`

### 7.3 可同步

条件：

- 找到配置目录
- 找到关键文件
- 最终规则下至少命中 1 项

## 8. 命令设计细化

## 8.1 `scan`

支持：

```text
ai-sync scan
ai-sync scan codex
ai-sync scan claude
ai-sync scan codex claude
```

设计决定：

1. 无参数时扫描全部支持工具。
2. 有参数时按用户传入顺序输出。
3. 当前只支持 `codex`、`claude`。
4. 如参数非法，CLI 直接报错。

## 8.2 `scan add`

支持：

```text
ai-sync scan add <app> <relative-file-path>
```

校验规则：

1. 必须是相对路径。
2. 不允许绝对路径。
3. 不允许 `..` 越界。
4. 目标必须存在。
5. 目标必须是文件。
6. 目标必须位于对应工具全局配置根目录下。

补充设计决定：

7. 如果路径已经在默认规则中，建议直接提示“该路径已属于默认同步范围”，不再重复保存为自定义规则。
8. 如果路径已存在于自定义规则中，按幂等成功处理或给出明确提示，不重复写库。

## 8.3 `scan remove`

支持：

```text
ai-sync scan remove <app> <relative-file-path>
```

设计决定：

1. 只允许删除用户自定义规则。
2. 不允许删除默认规则。
3. 若路径不存在于自定义规则，返回明确提示。

## 8.4 `scan list`

支持：

```text
ai-sync scan list
ai-sync scan list <app>
```

建议输出内容：

- 默认全局规则
- 默认项目规则
- 自定义全局规则

## 9. 对现有模块的改造建议

## 9.1 `internal/service/tool/paths.go`

改造方向：

- 保留“路径定义”职责
- 从旧的零散 definitions 过渡到统一规则定义
- 将 Claude `plugins/` 从目录规则改成 3 个文件规则

## 9.2 `internal/service/tool/detector.go`

改造方向：

- 不再只返回旧式 `ConfigFiles`
- 改为复用 `RuleResolver`
- 仍可保留“探测工具目录存在性”的壳层职责

## 9.3 `internal/service/tool/accessor.go`

改造方向：

- 允许访问范围改为“最终生效规则中的允许路径”
- 自定义规则必须自动生效

## 9.4 `internal/service/snapshot/collector.go`

改造方向：

- 不再按配置根目录整树遍历
- 改为遍历 `RuleResolver` 提供的最终命中文件/目录
- 目录型规则仅收集其目录下文件；文件型规则只收集单文件

## 9.5 `internal/app/usecase/local_workflow.go`

改造方向：

- `Scan` 增加 app 过滤输入
- 增加 `AddCustomRule`、`RemoveCustomRule`、`ListRules`
- `ToolSummary` 扩充为更适合 CLI/TUI 展示的结构

## 9.6 `internal/cli/cobra/scan.go`

改造方向：

- `scan` 从叶子命令改为带子命令的父命令
- 同时兼容 `scan [apps...]`
- 新增 `scan add/remove/list`

## 9.7 `internal/tui/page_scan.go`

改造方向：

- 显示中文状态
- 显示原因
- 显示可同步项数量
- 条件允许时显示提醒

## 10. 数据持久化设计

建议新增数据表：

```text
custom_sync_rules
```

建议字段：

- `id TEXT PRIMARY KEY`
- `tool_type TEXT NOT NULL`
- `scope TEXT NOT NULL`
- `relative_path TEXT NOT NULL`
- `created_at INTEGER NOT NULL`
- `updated_at INTEGER NOT NULL`

建议索引与约束：

- `UNIQUE(tool_type, scope, relative_path)`
- `INDEX(tool_type, scope)`

不建议复用 `app_settings.sync_includes`，原因如下：

1. 当前 `app_settings` 还没有成熟 DAO 与读写链路。
2. `sync_includes` 是聚合字符串，不适合做高质量增删查与唯一校验。
3. 本次命令需要独立、可验证、可测试的数据模型。

## 11. 风险与应对

## 11.1 风险：局部修改导致规则再次分叉

应对：

- 所有规则判断必须收口到 `RuleResolver`
- 禁止 CLI、accessor、collector 自己维护白名单

## 11.2 风险：`snapshot create` 对目录型规则打包不完整

应对：

- 明确区分“规则项命中”和“最终文件列表”
- 由解析层统一产出“可打包文件列表”

## 11.3 风险：`scan add` 后路径合法但文件随后被删除

应对：

- 规则保留
- `scan` 展示在“未命中自定义项”
- 不计入当前可同步数量

## 11.4 风险：CLI 与 TUI 再次展示不一致

应对：

- 统一 usecase 返回结构
- CLI/TUI 都只消费同一份 `ScanResult`

## 13. 最终建议

最终建议如下：

1. 采用“统一规则源”方案。
2. 使用 SQLite 新表保存自定义规则。
3. 将 Claude `plugins/` 默认规则改为 3 个文件级白名单。
4. 将关键文件门槛纳入统一状态判定。
5. 让 `scan`、`get`、`snapshot create` 全部改为依赖同一规则解析层。

如果只做表层命令增强，后续一定会继续出现以下问题：

- 展示和实际同步范围不一致
- `scan add` 与 `get` 断链
- 自定义规则无法稳定演进
- TUI 与 CLI 再次分叉

因此本次应按“统一规则收口”一次性完成主链路治理。
