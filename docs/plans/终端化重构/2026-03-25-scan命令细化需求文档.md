# AI Sync Manager `scan` 命令细化需求文档

## 文档信息

- 日期：2026-03-25
- 分类：`docs/plans/终端化重构/`
- 范围：`scan` 命令按应用过滤、Claude 插件目录细粒度同步、自定义全局同步规则、关键文件门槛判定

## 1. 背景

当前 `ai-sync scan` 存在以下问题：

1. 不能按应用扫描，只能固定扫描 `codex` 与 `claude`
2. Claude 的 `plugins/` 目录仍按整个目录识别，粒度过粗
3. 用户无法自行追加某个全局配置文件到同步范围
4. 当前“可同步”判定过宽，只要目录存在且命中任意项即可
5. `scan`、`get`、`snapshot create` 当前没有共享同一套最终同步规则

本次需求目标是统一这三条链路的规则来源，并把 `scan` 提升为“同步规则总入口”。

## 2. 目标

本次需求应实现：

1. 支持按 app 执行扫描
2. 支持 Claude 插件目录的默认白名单同步规则
3. 支持用户追加全局文件级自定义同步规则
4. 支持关键文件门槛判定
5. 统一 `scan`、`get`、`snapshot create` 的最终生效规则

## 3. 非目标

本次不做：

- 项目级自定义同步规则
- 自定义目录级规则
- 自定义不存在的占位路径
- 远端同步联动
- GUI 配置界面
- 自动从官方文档动态拉取规则

## 4. 命令设计

### 4.1 扫描命令

支持以下形式：

```text
ai-sync scan
ai-sync scan codex
ai-sync scan claude
ai-sync scan codex claude
```

规则如下：

- `scan` 无参数时扫描全部支持应用
- 有参数时，仅扫描指定 app
- 输出顺序按用户传入顺序展示
- app 名当前只支持 `codex`、`claude`

### 4.2 自定义规则命令

支持以下形式：

```text
ai-sync scan add <app> <relative-file-path>
ai-sync scan remove <app> <relative-file-path>
ai-sync scan list
ai-sync scan list <app>
```

规则如下：

- `scan add` 为指定 app 追加一个全局文件级自定义规则
- `scan remove` 删除该自定义规则
- `scan list` 查看当前规则
- `scan list <app>` 仅查看指定 app 的规则

## 5. 默认同步规则

### 5.1 Codex 默认规则

全局默认同步项：

- `config.toml`
- `skills/`
- `rules/`
- `superpowers/`

项目级默认同步项：

- `.codex/`
- `AGENTS.md`

### 5.2 Claude 默认规则

全局默认同步项：

- `settings.json`
- `CLAUDE.md`
- `commands/`
- `skills/`
- `output-styles/`
- `plugins/blocklist.json`
- `plugins/installed_plugins.json`
- `plugins/known_marketplaces.json`

项目级默认同步项：

- `.claude/`
- `CLAUDE.md`

规则说明：

- `plugins/` 不再整体视为默认同步目录
- `plugins/` 下除以上 3 个文件外的其他内容，默认不纳入同步

## 6. 关键文件门槛

### 6.1 Codex

Codex 必须识别到以下关键文件，才允许进入“可同步”状态：

```text
~/.codex/config.toml
```

### 6.2 Claude

Claude 必须识别到以下关键文件，才允许进入“可同步”状态：

```text
~/.claude/settings.json
```

## 7. 状态判定规则

`scan` 状态仍保留以下三类：

- `可同步`
- `暂不可同步`
- `不可同步`

判定优先级如下：

### 7.1 不可同步

满足以下条件时判定为 `不可同步`：

- 未找到对应应用的全局配置目录

### 7.2 暂不可同步

满足以下任一条件时判定为 `暂不可同步`：

- 找到配置目录，但缺少关键文件
- 找到配置目录和关键文件，但当前规则下未识别到任何可同步项

### 7.3 可同步

满足以下全部条件时判定为 `可同步`：

- 找到配置目录
- 找到关键文件
- 当前规则下至少识别到 1 项可同步配置

## 8. 错误与原因文案

状态为 `暂不可同步` 或 `不可同步` 时，必须输出具体原因。

建议原因文案如下：

- Codex 缺少关键文件：
  `缺少关键文件 ~/.codex/config.toml`
- Claude 缺少关键文件：
  `缺少关键文件 ~/.claude/settings.json`
- 配置目录存在但没有命中同步项：
  `找到了配置目录和关键文件，但当前规则下没有识别到可同步项`
- 配置目录不存在：
  `未找到配置目录 <path>`

## 9. 自定义全局同步规则

### 9.1 适用范围

第一阶段仅支持：

- 全局作用域
- 文件级规则

不支持：

- 项目级规则
- 目录级规则

### 9.2 添加规则约束

`scan add` 必须满足以下约束：

- 路径必须是相对于工具全局配置根目录的相对路径
- 不允许绝对路径
- 不允许 `..` 越界路径
- 目标必须是已存在的文件
- 不允许添加目录

### 9.3 规则持久化

自定义规则持久化到 AI Sync Manager 自己的数据目录，由程序维护。

要求：

- 不写回 `~/.codex` 或 `~/.claude`
- 不污染用户原始工具配置目录

### 9.4 文件删除后的行为

如果某条自定义规则对应文件后续被删除：

- 规则仍保留
- `scan` 中显示为“未命中自定义项”
- 不计入当前“可同步项”数量

## 10. 展示要求

### 10.1 CLI `scan`

CLI 输出至少包含：

- 扫描对象名称
- 检测结果
- 配置目录
- 可同步项数量
- 原因
- 该扫描对象下识别到的配置项

建议展示分组：

- `关键配置`
- `扩展内容`
- `其他内容`

### 10.2 Claude 插件目录提醒

当 `~/.claude/plugins/` 下存在未纳入默认白名单的其他文件时，建议增加提醒：

- `检测到未纳入默认同步范围的插件文件，后续可通过 scan add 补充`

### 10.3 TUI

TUI 扫描结果页至少需要改为显示：

- 扫描对象名
- 中文状态
- 原因
- 可同步项数量

要求：

- 不再只显示内部状态值 `installed / partial / not_installed`
- 与 CLI 的中文结果保持一致

## 11. 统一规则源要求

本次需求的核心约束是：

- `scan`
- `get`
- `snapshot create`

必须共享同一套最终生效规则。

### 11.1 对 `scan`

`scan` 负责：

- 识别命中项
- 展示默认规则与自定义规则
- 输出状态与原因

### 11.2 对 `get`

`get` 只允许访问最终生效规则允许的路径。

要求：

- 默认规则路径仍可访问
- 用户通过 `scan add` 增加的路径也必须可访问

### 11.3 对 `snapshot create`

`snapshot create` 只允许打包最终生效规则命中的文件。

要求：

- 不再对整个配置根目录无差别递归收集
- Claude `plugins/` 默认仅打包白名单 3 个文件
- 用户追加的自定义文件若存在，则一并打包

## 12. 验收标准

功能验收：

- `ai-sync scan codex` 仅输出 Codex
- `ai-sync scan claude` 仅输出 Claude
- `ai-sync scan codex claude` 按输入顺序输出
- Codex 缺少 `~/.codex/config.toml` 时显示 `暂不可同步`
- Claude 缺少 `~/.claude/settings.json` 时显示 `暂不可同步`
- Claude `plugins/` 默认仅识别 3 个白名单文件
- `scan add <app> <file>` 可以成功增加全局文件级规则
- `scan remove <app> <file>` 可以删除该规则
- `scan list [app]` 可以查看当前规则
- 自定义规则文件不存在时显示为“未命中自定义项”

链路一致性验收：

- `scan` 命中的自定义文件，`get` 可以访问
- `snapshot create` 只打包最终规则命中的文件
- CLI 与 TUI 的状态文案保持一致

文档验收：

- 使用指南同步更新 `scan` 的命令语法与状态定义
- 相关计划文档写入 `docs/plans/终端化重构/`

## 13. 实施建议

建议后续实施按以下顺序推进：

1. 抽取统一规则模型
2. 改造默认规则 definitions
3. 增加自定义规则存储与读写
4. 改造 detector 状态判定
5. 改造 CLI `scan` 命令与输出
6. 改造 accessor 白名单
7. 改造 snapshot collector
8. 改造 TUI 扫描页
9. 更新测试与使用文档

## 14. 结论

本次需求不是单一命令增强，而是同步规则体系的第一次收口。

后续实现必须以“统一规则源”为约束，否则会继续出现：

- `scan` 展示与 `snapshot create` 打包不一致
- `scan add` 后 `get` 无法访问
- CLI 与 TUI 展示不一致

因此该需求应作为终端化重构阶段中的一项规则层改造来实施。
