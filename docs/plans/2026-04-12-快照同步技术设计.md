## 2. 设计原则

- **SQLite 是唯一真相源** — 存储最新快照数据；工作目录是 Git 的 workspace，可从 SQLite 重建
- **Git 对用户完全透明** — 用户不直接接触任何 Git 命令
- **本地优先** — 没有 Git 远端仓库也能使用快照更新功能
- **Push 失败安全** — 网络失败不影响本地 SQLite 数据
- **冲突不自动覆盖** — 检测到冲突必须用户确认
- **git HEAD 作为同步基准** — 三方对比使用 git HEAD（上次 push 的版本）而非额外的 sync-state 文件

## 3. 架构总览

### 3.1 双重存储模型

```
┌─────────────────────────────────────────────┐
│              SQLite（真相源）                 │
│  snapshots + snapshot_files                  │
│  存储最新快照的完整文件内容                     │
└───────────────┬─────────────────────────────┘
                ↕ 同步（Push/Pull 时读写）
┌───────────────┴─────────────────────────────┐
│       Git 工作目录（版本引擎）                 │
│  ~/.fl/repos/<project>/                     │
│  文件内容与 SQLite 镜像，Git 负责版本历史       │
│  git HEAD = 上次 push 的版本                  │
└───────────────┬─────────────────────────────┘
                ↕ push/pull
┌───────────────┴─────────────────────────────┐
│       远端仓库（同步通道）                     │
│  GitHub / Gitea / 自建 Git                   │
└─────────────────────────────────────────────┘
```

### 3.2 数据流方向

- **Push**：SQLite → 工作目录（写文件）→ Git commit → Git push → 远端
- **Pull**：远端 → Git fetch → 对比变更 → 工作目录 → SQLite（三方对比决定是否写入）
- **Update**：磁盘文件 → Collector 收集 → SQLite（UpdateWithFiles）→ Git commit（可选）

### 3.3 当前仓库布局（独占模式）

```
repos/
├── claude-global/        # project 独占仓库
│   ├── .git/
│   └── claude-global/    # 项目子目录（保持原始路径结构）
│       ├── settings.json
│       └── .claude/CLAUDE.md
└── codex-global/         # 另一个 project 独占仓库
    ├── .git/
    └── codex-global/
        └── config.yaml
```

> **当前设计**：每个 project 独占一个 Git 仓库。未来如需共享仓库，需修改 push 时 `worktree.Add("project-name/")` 只 add 自己的子目录，pull 时只读写自己子目录的文件，不使用 ResetToRef。

## 5. 已实现：SyncPush

### 5.1 数据流

```
sync push --project claude-global
  │
  ├─ 1. 参数校验
  ├─ 2. 获取远端配置 → RemoteConfigDAO.ListRemoteConfigs()
  ├─ 3. 查找 project 最新快照 → SnapshotService.ListSnapshots/GetSnapshot（按 created_at DESC 排序，取第一个匹配项）
  ├─ 4. 确保工作目录存在
  │     已存在 → 跳过

  ├─ 5. 从 SQLite 读文件 → 写入工作目录

  ├─ 6. Git add + commit → GitClient.Commit(All: true)
  │     空提交 → 返回"无需推送"
  ├─ 7. Git push → GitClient.Push()
  ├─ 8. 验证远端 → GitClient.GetRemoteHeadHash() 对比本地 commit hash
  └─ 9. 返回 SyncPushResult
```

## 6. 已实现：SyncPull

### 6.1 数据流

```
sync pull --project claude-global
  │
  ├─ 1. 参数校验 + 获取远端配置
  ├─ 2. 检查本地仓库 → 不存在时报错"请先 push"
  ├─ 3. 记录本地 HEAD → GitClient.GetHeadHash()
  ├─ 4. Git fetch → GitClient.Fetch()
  ├─ 5. 获取远端 HEAD → GitClient.GetRemoteHeadHash()
  │     相同 → "本地已是最新"
  ├─ 6. 获取变更文件列表 → GitClient.GetChangedFiles(localHash, remoteHash)
  │     过滤只保留 project 子目录下的文件
  ├─ 7. 冲突检测（三方对比）
  │     for each changedFile:
  │       remoteContent = GitClient.GetFileContent(remoteHash)
  │       localSnapshotContent = 从 SQLite 读
  │       localGitContent = GitClient.GetFileContent(localHash)
  │
  │       if localSnapshot == localGit → 本地没改 → 安全
  │       if localSnapshot != localGit && remoteContent != localGit → 冲突
  ├─ 8a. 有冲突 → 返回冲突列表
  ├─ 8b. 无冲突 → GitClient.ResetToRef(remoteHash) + 更新 SQLite
  └─ 9. 返回 SyncPullResult
```

**重要记录：此处所用 "localSnapshot" 是** **最新创建的快照** **（按 created_at DESC 排序取第一个）。** **如果同一项目有多个快照，** **sync pull 只会和** **最新那个做对比和更新。** **其他旧快照不受影响，** **也不会被同步。**


### 6.2 三方对比详解

```
三方数据源：
  ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
  │ 本地快照 (SQLite) │     │ 本地 git HEAD    │     │ 远端 HEAD        │
  │ 用户当前数据      │     │ 上次 push 的版本  │     │ 远端最新版本      │
  └─────────────────┘     └─────────────────┘     └─────────────────┘
         A                       B                       C

判断逻辑：
  A == B && A != C → 用户没改，远端改了 → 安全（用远端 C）
  A != B && B == C → 用户改了，远端没改 → 不应出现在 diff 中（已 push 过）
  A != B && B != C → 用户改了，远端也改了 → 冲突！
  A == B && A == C → 两边都没改 → 无变化
```

> **为什么用 git HEAD 而非 sync-state.json**：git HEAD 天然可靠，不会丢失或损坏。每次 push 后 HEAD 自动更新，无需额外维护同步状态文件。消除了 sync-state 与 SQLite 不一致的风险。

### 6.3 关键代码位置

| 逻辑 | 文件 | 行号 |
|------|------|------|
| 入口 | `internal/app/usecase/sync_method.go` | 338-602 |
| Fetch + 获取变更 | `sync_method.go` | 391-445 |

### 6.4 ResetToRef 的使用

冲突检测通过后，使用 `GitClient.ResetToRef(remoteHash)` 将工作目录 hard reset 到远端版本。

```go
// 为什么用 ResetToRef 而不是 Pull：
// go-git 的 Pull 只支持 fast-forward merge，
// 当本地和远端有分叉历史时会报 "non-fast-forward update"。
// 已在第六步完成冲突检测，可以安全地 reset 到远端。
```

sync diff --project claude-global [-v] [--json] [--path filter] [--status modified]

  │     加 -v 时：
  │       - local_diff: 本地修改的内容片段

## 8. 待开发：冲突交互解决

### 8.1 数据流

```
用户执行 sync pull → 检测到冲突 → 返回冲突列表
  │
  ├─ 1. 展示冲突文件列表
  │     for each conflict:
  │       [N] path
  │           类型: both_modified
  │           本地: 已修改 (xxx 字节)
  │           远端: 已修改 (xxx 字节)
  │
  ├─ 2. 用户查看详细 diff（可选）
  │     展示本地修改和远端修改的内容片段
  │
  ├─ 3. 交互选择处理方式
  │     for each conflict:
  │       [1] 使用本地版本
  │       [2] 使用远端版本
  │       [3] 跳过此文件
  │       [4] 取消本次拉取
  │
  ├─ 4. 执行用户决策
  │     使用远端版本 → 从 git 读取远端文件内容 → 更新 SQLite
  │     使用本地版本 → 不动 SQLite，但需要确保工作目录文件正确
  │     跳过 → 不处理
  │
  ├─ 5. ResetToRef 到远端版本（更新工作目录）
  │     但保留"使用本地版本"的文件不变（工作目录层面）
  │     为什么：工作目录需要和远端保持一致，但 SQLite 中保留用户选择的版本
  │
  └─ 6. 返回处理结果汇总
```

### 8.2 接口定义

```go
// ResolveConflictsInput 冲突解决输入
type ResolveConflictsInput struct {
    Project  string             // 项目名称
    Decisions []ConflictDecision // 每个冲突文件的决策
}

// ConflictDecision 单个冲突文件的决策
type ConflictDecision struct {
    Path    string // 文件路径
    Action  string // "local" / "remote" / "skip"
}
```

### 8.3 实现策略

有两种方案：

**方案 A：单次交互（CLI prompt）**

pull 检测到冲突后，直接在 CLI 中逐文件询问用户，用户当场选择。不需要额外命令。

```
$ fl sync pull --project claude-global

检测到 2 个文件有冲突：

  [1] settings.json — 本地和远端都做了修改
  [2] .claude/CLAUDE.md — 本地和远端都做了修改

settings.json [1=本地 2=远端 3=跳过]: 2
.claude/CLAUDE.md [1=本地 2=远端 3=跳过]: 1

处理结果：
  settings.json    → 使用远端版本
  .claude/CLAUDE.md → 保留本地版本

已更新 1 个文件。
```

**方案 B：两步操作（pull 报告 + resolve 解决）**

```
$ fl sync pull --project claude-global
检测到 2 个冲突，请使用 sync resolve 解决

$ fl sync resolve --project claude-global
（进入交互模式）
```

> **推荐方案 A**。配置文件冲突通常只有 1-3 个文件，单次交互更直接。方案 B 增加了一个额外步骤，对用户负担更大。

## 11. 认证实现

### 11.1 支持的认证类型

| 类型 | 实现方式 | 说明 |
|------|---------|------|
| 无认证 | `nil` | 公开仓库 |
| Token | `http.BasicAuth{Username: "token", Password: token}` | GitHub PAT / GitLab Token |
| Basic | `http.BasicAuth{Username: user, Password: pass}` | 用户名密码 |
| SSH | `ssh.PublicKeys{Signer, User: "git"}` | 密钥文件（不支持 ssh-agent） |

### 11.2 SSH 密钥探测

`auth.go` 按以下顺序探测密钥：
1. 用户指定的密钥路径（`--ssh-key`）
2. 默认位置 `~/.ssh/id_ed25519`, `id_rsa`, `id_ecdsa` 等
3. 直接提供密钥内容（包含 "BEGIN" 的字符串）

### 11.3 已知限制

- `ssh.InsecureIgnoreHostKey()` — 跳过 host key 校验，存在 MITM 风险
- 不支持 ssh-agent — 用户必须指定密钥路径
- 不支持 Windows credential manager — Token 直接存 SQLite

## 12. 错误处理

| 场景 | 处理方式 | 代码位置 |

| 空提交 | 返回"无需推送" | `sync_method.go:209-231` |
| Push 后远端验证失败 | 报错"远端未收到提交" | `sync_method.go:301-319` |
