## 1. 背景与目标

### 1.1 现状

- 快照数据仅存储在本地 SQLite 数据库中，无法跨设备同步
- 快照创建后不支持更新，用户修改配置后只能新建快照
- 产品名 AToSync 但同步功能完全缺失

### 1.2 目标

基于 Git 构建透明的版本控制和同步引擎，实现：

- 快照更新能力（覆盖本地数据）
- 多设备配置同步（通过 Git 远端仓库）
- 版本历史管理（Git commit 记录）
- 冲突自动检测与用户引导处理（fetch + 三方对比，非 git merge）

### 1.3 核心理念

- **Git 对用户完全透明** — 用户不会直接接触任何 Git 命令
- **SQLite 是唯一真相源** — 存储最新快照数据（元数据 + 文件内容）
- **Git 是版本引擎** — 负责历史记录、冲突处理、远端同步
- **本地优先** — 没有 Git 远端仓库也能使用快照功能

### 1.4 技术决策记录

> 以下决策来自 2026-04-15 技术可行性评审。

| 决策 | 结论 | 理由 |
|------|------|------|
| go-git vs 系统 Git | 保持 go-git | 配置文件场景够用；Token 认证成熟；SSH 需用户指定密钥路径 |
| 三方对比基准 | 用 git HEAD 而非 sync-state.json | git commit 历史天然可靠，无需额外文件；消除 sync-state 丢失/不一致风险 |
| 共享仓库 | 当前每个 project 独占一个仓库 | 简单可靠，消除并发安全问题；未来可扩展支持共享仓库（子目录隔离） |
| Pull 策略 | Fetch + 对比 + ResetToRef | go-git 只支持 fast-forward merge，Fetch+Reset 绕过限制 |
| 首次 Pull | 支持自动 clone（不要求先 push） | 用户在新设备上使用的常见场景 |

## 3. 用户场景

### 场景 1：配置改了，更新快照

> 小明修改了 Claude 的 settings.json，执行 `fl snapshot update "日常配置"`，系统自动重新扫描并更新快照数据。旧版本通过 Git 历史保留。

### 场景 2：公司电脑推配置到家里电脑

> 小明在公司电脑执行 `fl sync push`，配置推送到 GitHub 仓库。回家后在家里的电脑执行 `fl sync pull`，配置自动同步。

### 场景 3：两台电脑都改了配置，有冲突

> 公司和家里都修改了 settings.json。家里电脑 pull 时系统检测到冲突，展示 diff 让小明选择保留哪个版本。

### 场景 4：多个工具推到同一个仓库

> 小红把 claude-global 和 codex-global 都推到同一个 GitHub 仓库，仓库里按项目名分目录，结构清晰。

> **注意**：当前版本每个 project 独占一个仓库。多 project 共享同一仓库的支持在未来版本中实现。

### 场景 5：回退到上周的配置

> 小刚发现最近的配置改动有问题，执行 `fl snapshot restore "日常配置" --history`，查看版本历史，选择上周的版本恢复。

### 场景 6：新设备首次拉取

> 小明买了一台新电脑，安装 fl 后执行 `fl remote add <url>` 添加远端仓库，然后执行 `fl sync pull --project claude-global`，自动 clone 远端仓库并同步配置。

### 场景 7：拉取前预览差异

> 小红想看看远端有没有人改了配置，执行 `fl sync diff --project claude-global`，看到远端修改了 2 个文件，确认后决定是否 pull。

## 4. 功能需求

### FR-01 快照更新

用户可以更新已有快照，用最新的配置文件覆盖数据库中的旧数据。

- 执行 update 时自动重新扫描关联 project 的配置文件
- 覆盖 SQLite 中的快照文件数据（元数据保留）
- 自动在工作目录执行 Git commit，保留历史版本
- update 前后如果文件内容无变化，提示"无变更"

### FR-02 远端仓库管理

用户可以配置 Git 远端仓库地址，作为同步通道。

- 支持添加远端仓库（remote add）
- 支持查看已配置的远端仓库（remote list）
- 支持删除远端仓库（remote remove）
- 支持 GitHub、Gitea、自建 Git 仓库
- 一个 project 绑定一个远端仓库
- 多个 project 可以共享同一个远端仓库（按子目录分开，未来版本支持）

### FR-03 推送到远端

用户可以将本地快照数据推送到远端 Git 仓库。

- 从 SQLite 读出快照最新数据
- 本地仓库不存在时自动 clone 远端仓库
- clone 已有远端仓库时，对比快照文件与远端已有文件，展示差异表格让用户确认
- 确认后写入到工作目录（`~/.fl/repos/<project-name>/`）
- 执行 git add + commit + push（对用户透明）
- 推送失败时展示错误信息，不影响本地数据
- 支持 `--force` 跳过差异确认直接推送
- push 成功后验证远端 commit hash 与本地一致

### FR-04 从远端拉取

用户可以从远端 Git 仓库拉取最新配置并更新本地数据库。

- 本地仓库不存在时自动 clone 远端仓库（首次使用场景）
- clone 后对比远端内容与本地快照，展示差异让用户确认
- 执行 git fetch（仅下载，不合并），对比本地 HEAD 与远端 HEAD
- 检测变更文件列表
- 三方对比检测冲突（本地快照 vs 本地 git HEAD vs 远端 HEAD）
- 无冲突时 hard reset 到远端版本，更新 SQLite 数据
- 有冲突时展示冲突列表，等待用户解决
- 拉取后提示用户"有 N 个文件更新"
- 支持 `--force` 跳过确认直接拉取

> **三方对比策略**：使用 git HEAD 作为基准（而非额外的 sync-state.json 文件）。本地 git HEAD 代表上次 push 的版本，本地快照代表当前数据。如果本地快照和 git HEAD 不同，说明用户本地修改过；如果远端也改了同一文件，则判定为冲突。

### FR-05 同步差异预览

在执行 pull 之前，用户可以预览远端与本地快照之间的差异，判断是否需要拉取。

- 执行 git fetch（仅下载不合并），获取远端最新 commit
- 对比本地 HEAD 与远端 HEAD，列出变更文件
- 默认显示文件列表级差异（新增 / 修改 / 删除），加 `-v` 显示内容差异
- 三方对比标记潜在冲突（本地和远端都修改的文件）
- 支持按项目、路径、变更类型过滤
- 支持 `--json` 输出供程序处理
- 纯只读操作，不修改本地任何数据

### FR-06 版本历史查看

用户可以查看某个快照的版本历史。

- 从 Git commit 日志读取历史记录
- 展示每次更新的时间、说明、变更文件数
- SQLite 可选存历史索引（commit hash、时间、说明）用于快速查询

### FR-07 历史版本恢复

用户可以恢复到快照的某个历史版本。

- 从 Git 历史中选择特定版本
- 展示 diff（当前版本 vs 历史版本）
- 用户确认后恢复（走快照恢复流程）

### FR-08 冲突交互解决（待开发）

当 sync pull 检测到冲突时，用户可以通过交互方式逐文件选择处理方式。

- 展示冲突文件列表（路径 + 冲突类型 + 本地/远端摘要）
- 对每个冲突文件，用户可选择：
  - 使用本地版本（保留本地，跳过远端变更）
  - 使用远端版本（用远端覆盖本地）
  - 跳过此文件（不处理）
  - 取消本次拉取
- 查看详细 diff（本地修改 vs 远端修改）
- 选择完成后，按用户决策更新 SQLite 和工作目录
- 处理完成后展示汇总（N 个文件使用远端、N 个保留本地、N 个跳过）

## 5. 数据架构

### 5.1 双重存储模型

```
┌─────────────────────────────────────────────┐
│              SQLite（真相源）                 │
│  ┌─────────┐  ┌──────────────┐              │
│  │ 快照元数据 │  │ 文件内容（最新） │              │
│  └─────────┘  └──────────────┘              │
│  ┌─────────────────────┐                    │
│  │ 历史索引（可选）       │ ← commit hash + 时间 + 说明  │
│  └─────────────────────┘                    │
└─────────────────────────────────────────────┘
                    ↕ 同步
┌─────────────────────────────────────────────┐
│          Git 仓库（版本引擎）                  │
│  ┌──────────────────────────────┐           │
│  │ 工作目录: ~/.fl/repos/   │           │
│  │  └── <project-name>/         │           │
│  │      ├── settings.json       │           │
│  │      └── .claude/CLAUDE.md   │           │
│  └──────────────────────────────┘           │
│  commit 历史: v1 → v2 → v3 → ...           │
│  git HEAD = 上次 push 的版本（三方对比基准）   │
└─────────────────────────────────────────────┘
                    ↕ push/pull
┌─────────────────────────────────────────────┐
│          远端仓库（同步通道）                  │
│  GitHub / Gitea / 自建 Git                   │
└─────────────────────────────────────────────┘
```

### 5.2 远端仓库与 project 关系

- 当前：一个 project 绑定一个远端仓库，独占一个本地 Git 仓库
- 未来：一个远端仓库可被多个 project 共用，每个 project 推送到不同的子目录
- 暂不支持一个 project 推到多个远端

## 6. CLI 命令

### 6.1 快照更新

```
fl snapshot update <id-or-name>           # 更新快照（自动重新扫描）
fl snapshot update <id-or-name> -m "说明"  # 更新时附加说明
fl snapshot history <id-or-name>          # 查看版本历史
```

### 6.2 远端仓库管理

```
fl remote add <url>                       # 添加远端仓库
fl remote add <url> --project <name>      # 添加并绑定到指定 project
fl remote list                            # 查看已配置的远端仓库
fl remote remove <name>                   # 删除远端仓库
```

### 6.3 同步操作

```
fl sync push                              # 推送当前 project 到远端
fl sync push --project <name>             # 推送指定 project
fl sync push --all                        # 推送所有 project
fl sync push --force                      # 跳过差异确认，直接推送
fl sync pull                              # 从远端拉取当前 project
fl sync pull --project <name>             # 拉取指定 project
fl sync pull --all                        # 拉取所有 project
fl sync pull --force                      # 跳过确认，直接拉取
fl sync diff                              # 预览远端与本地差异
fl sync diff --project <name>             # 预览指定项目的差异
fl sync diff -v                           # 显示内容级差异
fl sync diff --json                       # JSON 格式输出
fl sync diff --path <pattern>             # 按路径过滤（支持通配符）
fl sync diff --status modified            # 按变更类型过滤（added/modified/deleted）
fl sync status                            # 查看同步状态
fl sync status --project <name>           # 查看指定项目的同步状态
```

### 6.4 历史版本恢复

```
fl snapshot restore <id-or-name> --history        # 交互式选择历史版本
fl snapshot restore <id-or-name> --version <hash>  # 恢复指定版本
```

## 7. 交互流程

### 7.1 快照更新

```
用户执行 snapshot update
       ↓
  找到快照？──否──→ 报错："快照 'xxx' 不存在"
       ↓ 是
  自动重新扫描关联 project 的配置文件
       ↓
  内容有变化？──否──→ 提示"无变更"
       ↓ 是
  更新 SQLite 数据
       ↓
  自动 Git commit（透明）
       ↓
  展示更新结果：N 个文件已更新
```

### 7.2 推送到远端

```
用户执行 sync push
       ↓
  已配置远端仓库？──否──→ 提示"请先执行 remote add"
       ↓ 是
  从 SQLite 读出最新快照数据
       ↓
  本地仓库已存在？──否──→ 尝试 clone 远端仓库
       │                     ├─ clone 成功（远端已有内容）
       │                     │   ↓
       │                     │   对比快照文件 vs 远端已有文件
       │                     │   ↓
       │                     │   有差异？──否──→ 直接进入 commit + push
       │                     │   ↓ 是
       │                     │   展示差异表格（将被覆盖/新增/删除的文件）
       │                     │   ↓
       │                     │   用户确认？──否──→ 取消推送
       │                     │   ↓ 是
       │                     │   写入快照文件到工作目录
       │                     │
       │                     └─ clone 失败（空仓库/网络不通）
       │                         ↓
       │                         初始化新仓库，直接写入
       ↓ 是
  写入工作目录（~/.fl/repos/<project>/）
       ↓
  Git add + commit + push（透明）
       ↓
  验证远端 commit hash 与本地一致
       ↓
  展示推送结果：N 个文件已推送
```

> **设计决策**：push 首次 clone 已有远端仓库时，必须展示差异并让用户确认。避免在不知情的情况下覆盖远端已有内容。`--force` flag 可跳过确认。

### 7.3 从远端拉取

```
用户执行 sync pull
       ↓
  已配置远端仓库？──否──→ 提示"请先执行 remote add"
       ↓ 是
  已有本地仓库？──否──→ 自动 clone 远端仓库
       │                     ├─ clone 成功
       │                     │   ↓
       │                     │   对比远端文件 vs 本地快照（如果有）
       │                     │   ↓
       │                     │   展示差异表格（新增/修改/删除的文件）
       │                     │   ↓
       │                     │   用户确认？──否──→ 取消拉取
       │                     │   ↓ 是
       │                     │   将远端内容写入 SQLite
       │                     │   ↓
       │                     │   展示"N 个文件已同步"
       │                     │
       │                     └─ clone 失败 → 报错"无法连接远端仓库"
       ↓ 是
  Git fetch（仅下载，不合并）
       ↓
  对比本地 HEAD 和远端 HEAD → 获取变更文件列表
       ↓
  无变更？──是──→ 提示"本地已是最新"
       ↓ 否
  三方对比（本地快照内容 vs 本地 git HEAD 内容 vs 远端 HEAD 内容）
       ↓
  有冲突？──否──→ Hard reset 到远端版本 → 更新 SQLite → 展示"N 个文件已更新"
       ↓ 是
  展示冲突文件列表（路径 + 冲突类型 + 本地/远端摘要）
       ↓
  交互解决冲突（FR-08）或提示用户手动处理
```

> **三方对比原理**：本地 git HEAD 的文件内容代表"上次 push 的版本"。如果本地快照（SQLite）中的文件内容和 git HEAD 中的不同，说明用户在本地修改了配置。如果远端同一文件也改了，则判定为冲突。

> **设计决策**：pull 在本地仓库不存在时自动 clone 远端，不再强制要求先 push。这是首次在新设备上使用的常见场景。

### 7.4 同步差异预览

```
用户执行 sync diff
       ↓
  已配置远端仓库？──否──→ 提示"请先执行 remote add"
       ↓ 是
  已有本地仓库？──否──→ 自动 clone 远端仓库
       │                     ├─ clone 成功 → 展示远端全部文件 vs 本地快照的差异
       │                     └─ clone 失败 → 报错"无法连接远端仓库"
       ↓ 是
  Git fetch（仅下载，不合并）
       ↓
  对比本地 HEAD 和远端 HEAD → 获取变更文件列表
       ↓
  无变更？──是──→ 提示"本地已是最新"
       ↓ 否
  三方对比，标记冲突文件
       ↓
  展示差异表格：

  ┌────────────────────┬──────────┬───────────────────────────┐
  │ 文件路径            │ 变更类型  │ 冲突状态                  │
  ├────────────────────┼──────────┼───────────────────────────┤
  │ settings.json      │ modified │ 冲突（本地和远端都修改）   │
  │ CLAUDE.md          │ modified │ 安全（仅远端修改）         │
  │ plugins/config.json│ added    │ 安全（远端新增）           │
  └────────────────────┴──────────┴───────────────────────────┘

  加 -v 时追加内容差异（类 git diff 输出）
```

## 9. 错误处理

| 场景 | 处理方式 |
|------|---------|
| 快照不存在 | 报错："快照 'xxx' 不存在" |
| 未配置远端仓库 | 提示："请先执行 fl remote add <url>" |
| 远端仓库不可达 | 报错："无法连接远端仓库，请检查网络和仓库地址" |
| Push 失败 | 报错展示具体原因，本地数据不受影响 |
| Pull 有冲突 | 展示冲突文件列表，等待用户解决 |
| Git 操作失败 | 展示错误信息，建议用户检查工作目录 |
| 工作目录不存在 | 自动 clone 远端仓库；clone 失败则初始化新仓库 |
| 认证失败 | 提示检查 Token/SSH 密钥配置 |
| Push 后远端验证失败 | 报错"远端未收到提交"，本地数据不受影响 |

## 10. 非功能需求

### NFR-01 安全性

- Git 凭证使用系统原生 Git 凭证管理（credential helper）
- Token/SSH 密钥路径存储在本地 SQLite 数据库中，不以明文日志输出
- 敏感字段加密存储（后续阶段接入 AES-256-GCM 加密）
- SSH 连接使用 `ssh.InsecureIgnoreHostKey()`（已知限制，未来需改为 known_hosts 校验）

### NFR-02 性能

- 快照更新（100 个文件）在 3 秒内完成
- Push/Pull 操作主要耗时在网络传输，本地处理 < 2 秒
- 历史查询响应 < 1 秒

### NFR-03 兼容性

- 支持 Windows、macOS、Linux
- 支持 HTTP/HTTPS/SSH 协议的 Git 仓库
- 兼容 GitHub、Gitea、GitLab、自建 Git 服务
- go-git 不支持 ssh-agent，SSH 场景需用户指定密钥路径

### NFR-04 可靠性

- Push 失败不影响本地 SQLite 数据
- Pull 冲突不自动覆盖，必须用户确认
- 工作目录损坏可从 SQLite 重建
- Push 后验证远端 commit hash 与本地一致
