# GORM 数据库迁移 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将项目中的数据库访问从 `database/sql` 全量切换到 GORM，同时保持现有 SQLite 数据文件与表结构兼容。

**Architecture:** 以 `pkg/database` 为数据库底座迁移入口，将连接、事务和迁移统一改为 GORM；`internal/models/*DAO` 逐个切换为 GORM CRUD；SQLite 触发器、部分索引和兼容性 DDL 通过 `gorm.DB.Exec(...)` 保底执行，避免旧库 schema 漂移。

**Tech Stack:** Go 1.25, GORM, SQLite, testify

---

## 实施说明

- 先阅读设计文档：`docs/plans/2026-04-03-gorm数据库迁移设计.md`
- 本次迁移的第一原则是“兼容旧库”，不是“尽量少写 SQL”
- 不允许在业务 DAO 中继续直接使用 `database/sql`
- 允许在数据库初始化和兼容性迁移层保留少量 SQLite DDL SQL，但执行入口必须是 GORM
- 每完成一组 DAO 迁移，就立即运行对应测试，不要堆积到最后

### Task 1: 引入 GORM 依赖并改造数据库底座

**Files:**
- Modify: `go.mod`
- Modify: `go.sum`
- Modify: `pkg/database/db.go`

**Step 1: 写数据库初始化失败测试**

在数据库测试中增加一条约束：

- `InitTestDB` 返回的实例必须持有可用的 GORM 连接
- 初始化后仍能获取数据库文件路径

**Step 2: 运行测试确认当前实现不满足新约束**

Run: `go test ./pkg/database -run TestInitTestDBGormReady -count=1`

Expected: FAIL，因为当前数据库层仍基于 `*sql.DB`

**Step 3: 添加 GORM 依赖**

在 `go.mod` 中引入：

- `gorm.io/gorm`
- `gorm.io/driver/sqlite`

不要更换底层数据库类型，仍然使用 SQLite 文件。

**Step 4: 重写 `pkg/database/db.go`**

实施要点：

- `DB.conn` 改为 `*gorm.DB`
- 初始化时通过 GORM 打开 SQLite
- 从 GORM 获取底层 `sql.DB` 来设置连接池参数
- `PRAGMA foreign_keys = ON` 改为通过 `gorm.DB.Exec(...)`
- `Transaction` 改为 `func (db *DB) Transaction(fn func(*gorm.DB) error) error`

**Step 5: 保留兼容性迁移入口**

不要删除迁移职责，而是重构为：

- 基础结构迁移
- 索引补齐
- trigger 补齐

其中 trigger 和关键索引继续通过 `Exec` 执行。

**Step 6: 统一生产库和测试库初始化逻辑**

`InitDB` 与 `InitTestDB` 必须走同一套 GORM 初始化路径。

**Step 7: 运行数据库测试**

Run: `go test ./pkg/database/... -count=1`

Expected: PASS

**Step 8: Commit**

```bash
git add go.mod go.sum pkg/database/db.go
git commit -m "refactor: migrate database core to gorm"
```

## 风险提醒

- 不要在迁移过程中顺手重做领域模型
- 不要把“旧库兼容”让位给“GORM 默认行为”
- 复杂 DAO 必须在简单 DAO 稳定后再动
- 每组变更都要先有失败测试，再写最小实现
