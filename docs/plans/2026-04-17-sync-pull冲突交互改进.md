# sync pull 冲突交互改进 PRD

> 创建时间：2026-04-17
> 状态：待开发
> 优先级：P1
> 前置依赖：快照同步功能（sync pull 基本流程）

---

## 1. 背景与目标

### 1.1 现状

sync pull 检测到冲突时存在以下问题：

1. **差异不可见**：冲突文件只显示文件名和字节数，用户无法看到具体内容差异，无法做出明智选择
2. **交互不统一**：两个冲突场景（三方冲突 / 快照 vs 远端不一致）的交互方式不同，场景 2 是全量选择（所有文件统一决策）
3. **冗余选项**：菜单中 [3] 跳过与 [2] 使用远端效果完全一样
4. **单侧变更静默处理**：远端新增文件或本地独有文件被静默处理，用户无感知

### 1.2 目标

- 冲突文件支持查看具体内容差异（unified diff 格式），帮助用户做出明智决策
- 统一两个场景的交互方式，都支持逐文件处理
- 移除冗余的 [3] 跳过选项，替换为查看差异
- 单侧新增/删除自动解决并详细告知用户

---

## 2. 非目标

- 不做自动合并（不做内容级别的自动合并）
- 不做并排对比（只支持 unified diff）
- 不做结构化差异（不做 JSON/YAML key 级别的智能对比）

---

## 3. 冲突场景说明

### 3.1 场景 1：三方冲突（localHash != remoteHash）

远端有人 push 了新内容，git 层面有变更。三方对比：

```
本地快照(SQLite)  vs  本地Git HEAD(上次push)  vs  远端Git HEAD(最新fetch)
```

- 本地快照 == 本地 Git HEAD → 本地没改过 → 安全合并
- 本地快照 != 本地 Git HEAD 且远端也有修改 → 冲突

### 3.2 场景 2：快照 vs 远端不一致（localHash == remoteHash）

Git 层面没有变更（远端没人 push），但本地 SQLite 快照和远端仓库文件内容不同。

典型场景：首次 clone 了一个已有配置的仓库。

## 4. 功能详情

### 4.1 统一冲突菜单

两个场景统一为相同的交互菜单：

```
[1] 使用本地版本（保留本地修改）
[2] 使用远端版本（用远端覆盖）
[3] 查看差异
[4] 取消本次拉取
```

- 默认值：**2**（使用远端版本），**每次提示时默认值都是 2**，包括查看 diff 后的重新提示
- 选 [3] 时展示 unified diff，然后**重新提示选择**（可重复查看，每次默认值均为 2）
- 移除原来的 [3] 跳过选项（与 [2] 效果相同）

### 4.2 查看差异功能

选 [3] 时展示完整 unified diff：

```
  settings.json [1-4](默认=2): 3

  --- 本地 (599B) → 远端 (1655B) ---
  @@ -1,5 +1,12 @@
   {
  -  "theme": "dark"
  -  "fontSize": 14
  +  "theme": "light"
  +  "fontSize": 16
  +  "language": "zh-CN"
  +  "autoSync": true
  +  "plugins": ["git", "go"]
   }

  settings.json [1-4](默认=2):
```

- 使用 unified diff 格式（`+`/`-` 前缀），复用现有的 `RenderUnifiedDiff`
- **全量展示**，不做截断
- 二进制文件提示"[二进制文件，大小: X → Y 字节]"，不展示 diff
- 冲突文件列表显示行级差异统计（如 `+12/-3 行`）

### 4.3 单侧变更自动解决

单侧新增/删除**不视为冲突**，自动解决但详细告知用户：

| 情况 | 处理方式 | 用户提示 |
|------|---------|---------|
| 远端有，本地无 | 自动添加到本地快照 | `远端新增文件 xxx（Y 字节），本地不存在此文件，已自动同步到本地快照` |
| 本地有，远端无 | 保留本地版本不变 | `本地快照有文件 xxx，远端已无此文件，保留本地版本不变` |

**边界情况说明**：

- **双方修改但 hash 相同**：不可能出现。hash 基于 SHA256，只有内容完全一致时 hash 才相同，此时内容也相同，不会被视为冲突
- **本地新增未 push 的文件在场景 1 中的归类**：此类文件存在于本地快照但不存在于本地 Git HEAD（因为从未 push）。如果远端也未修改此文件（远端文件与本地 Git HEAD 一致），则视为"本地有修改但远端没改"→ 安全合并（保留本地版本）。如果远端也新增了同名文件，则进入正常三方冲突流程

展示格式：

```
自动同步（无冲突）：
  - new-plugin.yaml — 远端新增文件（320 字节），本地不存在此文件，已自动同步到本地快照
  - old-config.json — 本地快照有文件，远端已无此文件，保留本地版本不变
```

### 4.4 场景 2 逐文件交互

将场景 2 从全量三选一改为**逐文件交互**：

- 分类：遍历远端文件 + 遍历本地快照，区分自动解决 / 真实冲突
- 自动解决的文件打印通知
- 真实冲突文件逐个展示菜单（同 4.1）
- 支持查看差异（同 4.2）

## 5. 技术方案

### 5.1 新建共享 diff 工具包

**目录**：`internal/util/diffutil/`（新建目录）

**文件**：`internal/util/diffutil/diff.go`（新建）

**依赖关系说明**：本包仅依赖 `gotextdiff` + `myers`（已在 go.mod）和 `internal/types/snapshot`（纯类型定义）。**不依赖** `internal/service/snapshot/service.go`，因此不存在循环依赖。`service.go` 改为导入 `diffutil`，依赖方向为单向：`service.go` → `diffutil`。

从 `internal/service/snapshot/service.go` **搬迁**以下函数为导出函数（搬迁后从 service.go 中删除）：

| 原函数 | 新函数 | 原位置 |
|--------|--------|--------|
| `computeFileHunks` | `ComputeFileHunks` | service.go:892-901 |
| `parseUnifiedDiffLines` | `ParseUnifiedDiffLines` | service.go:904-957 |
| `countLineChanges` | `CountLineChanges` | service.go:1017-1031 |

新增组合函数：

```go
// ComputeConflictDiff 从本地和远端内容生成 DiffResult，供冲突 diff 展示使用。
func ComputeConflictDiff(localContent, remoteContent []byte, filePath string) *typesSnapshot.DiffResult
```

- 调用 `ComputeFileHunks` 生成 hunks
- 统计 AddLines/DelLines
- 二进制文件只填 IsBinary + OldSize/NewSize
- 依赖 `gotextdiff` + `myers`（已在 go.mod）

func (w *LocalWorkflow) showConflictDiff(conflict typesSync.ConflictInfo) {

#### 5.3.2 重写 `resolveConflicts`

**函数签名**（保持与现有调用方兼容）：

- 选 [3] 时调用 `showConflictDiff`，`continue` 重新提示
- 冲突文件展示附带行级统计

#### 5.3.3 场景 1 冲突检测调整

- `ConflictInfo` 填充 `LocalContent`、`RemoteContent`、`IsBinary`
- 单侧新增文件记入 `autoResolved` 而非静默 `safeFiles`
- 进入交互前打印自动解决通知

#### 5.3.4 新增 `classifySnapshotDifferences`

**函数签名**：

```go
func (w *LocalWorkflow) classifySnapshotDifferences(
    repoPath string,
    remoteHash string,
    localSnapshot *models.Snapshot,
    projectPrefix string,
) (conflicts []typesSync.ConflictInfo, autoResolved []typesSync.AutoResolvedInfo, err error)
```

替代原有的 `compareSnapshotWithRemoteSub`，**迁移路径**：

1. 原 `compareSnapshotWithRemote`（入口函数）和 `compareSnapshotWithRemoteSub`（递归遍历）的整体逻辑合并到 `classifySnapshotDifferences`
2. 保留递归遍历远端工作目录的能力（深度限制 depth=10）
3. 新增：遍历本地快照文件，用 `map[string]bool` 记录远端已有的路径，找出本地独有的文件
4. 原函数返回 `[]ConflictInfo`（不区分类型），新函数返回 `(conflicts, autoResolved)` 二元组
5. 原 `compareSnapshotWithRemote` 和 `compareSnapshotWithRemoteSub` 在迁移后删除

具体逻辑：
- 遍历远端工作目录文件，与本地快照逐个对比
- 遍历本地快照文件，检查哪些在远端不存在（原逻辑未覆盖）
- `remote_new`（远端有本地无）→ autoResolved
- `local_only`（本地有远端无）→ autoResolved
- `both_modified`（双方都有且内容不同）→ conflicts，同时填充 `LocalContent`/`RemoteContent`/`IsBinary`

#### 5.3.5 重写场景 2 交互

从全量选择改为逐文件处理：
1. 调用 `classifySnapshotDifferences`
2. 打印自动解决通知
3. 调用 `resolveConflicts` 逐文件交互
4. 合并用户选择 + 自动解决结果，更新 SQLite

## 8. 验收标准

### 8.1 正向测试

| 测试场景 | 预期结果 |
|---------|---------|
| 场景 1 三方冲突，选 [3] 查看 diff | 显示完整 unified diff，可重复查看 |
| 场景 1 三方冲突，选 [1] 保留本地 | 本地快照不变，工作目录写入本地版本 |
| 场景 1 三方冲突，选 [2] 使用远端 | 远端内容覆盖本地快照 |
| 场景 1 三方冲突，选 [4] 取消 | 不做任何变更 |
| 远端新增文件，本地不存在 | 自动同步，显示详细通知 |
| 本地独有文件，远端不存在 | 保留本地，显示详细通知 |
| 场景 2 逐文件交互 | 每个冲突文件单独选择，支持查看 diff |
| 二进制文件冲突 | 提示大小差异，不尝试展示 diff |
| `go test ./...` | 全部通过 |

### 8.2 负向测试

| 测试场景 | 预期结果 |
|---------|---------|
| 输入非法选项（如输入 5、abc） | 提示"无效输入，请重新选择"，不退出、不崩溃 |
| 多次输入非法选项后输入正确选项 | 正常处理，无副作用 |

| 无冲突文件时 sync pull | 不展示冲突菜单，直接完成同步 |
| 冲突文件内容完全相同（仅元数据不同） | 不视为冲突，走安全合并路径 |
