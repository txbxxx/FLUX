# AI Sync Manager Frontend Roadmap Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Extend the current homepage-based frontend into a usable desktop product flow by implementing homepage actions, snapshot management, repository setup, sync operations, and settings in a staged, test-driven sequence.

**Architecture:** Continue using the current Vue 3 + Vue Router desktop shell and keep state local to each page through small composables and API adapters. Build the product flow in vertical slices: each stage should ship a usable page, wire real Wails APIs, add the minimum test coverage, and leave the next stage with a stable UI shell instead of placeholder text.

**Tech Stack:** Vue 3, TypeScript, Vue Router, Vitest, @vue/test-utils, Wails 2 generated bindings, Go backend APIs

- Homepage shell is already implemented and polished.

## Delivery Sequence

1. Homepage action wiring and scan feedback
2. Snapshot page MVP
3. Repository page MVP
4. Sync page MVP
5. Settings page MVP
6. Global polish and integration verification

## Execution Rules

- Use TDD for every behavior change.
- Do not manually edit generated files under `frontend/wailsjs/`.
- Prefer page-scoped composables over premature global state.
- Keep each stage independently shippable.
- Commit after each task completes and verification passes.

## Risks To Watch

- Backend repository APIs are not yet complete for full remote setup, so the repository page must not promise unsupported behavior.
- Some backend methods may return partial or environment-specific data, so page composables should prefer graceful degradation over page crashes.
- Generated bindings may change if backend signatures move, so regenerate them before wiring any newly added API.
- Snapshot, sync, and encryption flows all touch stateful backend services, so manual `wails dev` verification is required even after tests pass.
