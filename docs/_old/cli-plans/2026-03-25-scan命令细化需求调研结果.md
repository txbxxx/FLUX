# AI Sync Manager `scan` 命令细化需求调研结果

## 文档信息

- 日期：2026-03-25
- 分类：`docs/plans/终端化重构/`
- 关联主题：`scan` 命令按应用过滤、Claude 插件细粒度同步、自定义全局同步规则、关键文件门槛判定

## 1. 调研范围

本次调研覆盖以下内容：

- 现有 `scan` 命令、workflow、detector、snapshot collector 的实现
- 当前 `get` 命令对白名单路径的访问控制
- CLI / TUI 的扫描结果展示
- 现有使用文档与计划文档
- Claude 官方插件文档

重点目标是确认：

1. 当前能力与用户新需求之间的差距
2. 哪些模块会被连带影响
3. 是否存在隐性遗漏或规则冲突

## 2. 调研结论摘要

结论如下：

1. 当前 `scan` 不支持按应用过滤，只能固定扫描 `codex` 与 `claude`。
2. 当前“可同步 / 暂不可同步 / 不可同步”判定只依赖“配置目录是否存在”和“是否命中任意配置项”，还没有“关键文件门槛”机制。
3. 当前 Claude 的 `plugins/` 会被整体视为一个可同步目录，不符合“只同步 3 个指定文件”的新需求。
4. 当前 `snapshot create` 与 `scan` 使用的不是同一套规则。`scan` 只展示识别结果，`snapshot create` 仍会递归收集整个配置根目录，存在展示与实际同步范围不一致的问题。
5. 当前 `get` 命令访问白名单来自静态 definitions。如果后续支持 `scan add`，但不同步扩展白名单，用户会遇到“scan 看得到但 get 读不到”的问题。
6. 当前 TUI 扫描页仍展示内部状态值，未展示中文结果、原因和细粒度配置项。

## 3. 现状核对结果

### 3.1 `scan` 命令入口

当前 `scan` 是无参数叶子命令，直接调用 `Workflow.Scan(ctx)`，没有任何 app 过滤能力。

相关代码：

- `internal/cli/cobra/scan.go`
- `internal/cli/cobra/root.go`

现状影响：

- 无法执行 `ai-sync scan codex`
- 无法执行 `ai-sync scan claude`
- CLI 帮助信息中也没有体现可选 app 参数

### 3.2 usecase 层行为

当前 `LocalWorkflow.Scan` 固定以全局扫描方式调用 detector，并且固定返回两个工具摘要：

- `codex`
- `claude`

相关代码：

- `internal/app/usecase/local_workflow.go`

现状影响：

- 即使未来 CLI 支持传参，usecase 层也还需要增加“仅扫描指定应用”的输入能力
- 当前始终返回两个工具，不适合实现“只输出指定应用”

### 3.3 状态判定逻辑

当前 detector 的判定规则是：

- 配置目录不存在 -> `not_installed`
- 配置目录存在且命中至少一项定义文件 -> `installed`
- 配置目录存在但未命中任何定义文件 -> `partial`

相关代码：

- `internal/service/tool/detector.go`

现状影响：

- Codex 即使没有 `~/.codex/config.toml`，只要目录下有别的匹配项，仍可能被判定为“可同步”
- Claude 即使没有 `~/.claude/settings.json`，只要 `skills/` 或 `commands/` 存在，也可能被判定为“可同步”
- 这与新需求“关键文件必须存在才可同步”不一致

### 3.4 默认扫描规则定义

当前默认定义如下。

Codex 全局：

- `config.toml`
- `skills/`
- `rules/`
- `superpowers/`

Claude 全局：

- `skills/`
- `commands/`
- `plugins/`
- `output-styles/`
- `CLAUDE.md`
- `settings.json`

相关代码：

- `internal/service/tool/paths.go`

现状影响：

- Claude `plugins/` 当前是目录级规则，不是文件级白名单规则
- 要满足新需求，至少要把 `plugins/` 从“整个目录”改成指定 3 个文件

### 3.5 `snapshot create` 收集规则

当前快照收集逻辑不是按 detector 识别出的“可同步项”执行，而是直接从工具配置根目录递归收集全部文件。

相关代码：

- `internal/service/snapshot/collector.go`
- `internal/service/snapshot/service.go`

现状影响：

- 如果只修改 `scan`，不修改 collector，会出现严重错配：
  - `scan` 可能显示 Claude 插件只同步 3 个文件
  - `snapshot create` 实际却把 `~/.claude/plugins/` 下全部文件打进快照
- 因此本次需求必须把 `scan` 与 `snapshot create` 的规则统一

### 3.6 `get` 命令路径白名单

当前 `get` 命令的允许访问路径来自静态 definitions，并且只基于全局规则白名单。

相关代码：

- `internal/service/tool/accessor.go`

现状影响：

- 如果未来用户通过 `scan add claude xxx.json` 添加了新文件
- 但 accessor 白名单未同步更新
- 则会出现“scan 识别到了，但 get 无法读取”的冲突

结论：

- 自定义规则不能只作用于 `scan`
- 还必须进入 `get` 的允许访问范围

### 3.7 TUI 扫描结果页

当前 TUI 扫描页仅显示：

- 工具名
- 内部状态值
- 配置数量

相关代码：

- `internal/tui/page_scan.go`

现状影响：

- 不显示中文状态
- 不显示具体原因
- 不显示细粒度可同步项
- 与 CLI 输出已经明显不一致

### 3.8 现有文档

当前使用指南中对 `scan` 的描述仍基于旧规则，状态说明为：

- 可同步：已找到配置目录，并识别到至少一项可同步配置
- 暂不可同步：找到了配置目录，但当前规则下没有识别到可同步项
- 不可同步：没找到对应工具的配置目录

相关文档：

- `docs/使用指南/命令行与终端界面MVP使用指南.md`

现状影响：

- 一旦引入关键文件门槛，状态定义与错误原因都要同步更新

## 4. Claude 官方文档核对结果

参考文档：

- <https://code.claude.com/docs/zh-CN/plugins>

核对结论：

1. 官方插件文档明确描述了将 `commands`、`agents`、`skills` 和 hooks 迁移到插件体系的思路。
2. 官方页面并未直接写明“插件迁移时只需要同步 `blocklist.json`、`installed_plugins.json`、`known_marketplaces.json` 这 3 个文件”。

因此本项目中的这 3 个文件规则，应定义为：

- AI Sync Manager 针对 Claude 插件目录的默认同步白名单

而不建议写成：

- Claude 官方迁移步骤明确要求只同步这 3 个文件

## 5. 用户确认结果

本轮已确认的产品边界如下：

1. `scan add` 第一阶段只做全局规则，不做项目级自定义规则。
2. `scan add` 只允许添加已存在的全局文件，不允许登记不存在的占位路径。
3. 自定义规则持久化到 AI Sync Manager 自己的数据目录，不写回 `~/.codex` 或 `~/.claude`。
4. `scan` 无参数默认扫描全部支持应用；有参数时按指定 app 输出。
5. `snapshot create` 后续必须严格跟随最终规则清单，不再递归整个配置根目录。
6. 自定义规则指向的文件后续如果不存在，规则仍保留，并在 `scan` 中显示为“未命中自定义项”。

## 6. 识别出的遗漏点

除用户显式提出的需求外，还识别出以下遗漏点：

### 6.1 `scan add` 需要配套命令

如果只支持 `scan add`，则规则无法查看和删除。

建议同时补充：

- `scan remove <app> <relative-file-path>`
- `scan list [app-or-project]`
- `scan rules [app-or-project]`

### 6.2 状态判定需要明确优先级

建议优先级如下：

1. 未找到配置目录 -> `不可同步`
2. 找到配置目录但缺少关键文件 -> `暂不可同步`
3. 找到关键文件但没有任何命中的同步项 -> `暂不可同步`
4. 找到关键文件且至少命中 1 项同步项 -> `可同步`

### 6.3 需要区分“默认规则命中项”和“用户自定义命中项”

否则 `scan add` 之后用户无法判断哪些是系统默认同步内容，哪些是自己追加的内容。

### 6.4 需要展示“未命中自定义项”

因为用户已明确要求：

- 文件后来不存在时，规则保留
- `scan` 应显示为“未命中自定义项”

### 6.5 需要给 Claude 插件目录的未纳入项增加提醒

由于默认仅同步 3 个白名单文件，建议当 `plugins/` 下还有其他文件时增加提醒，例如：

- “检测到未纳入默认同步范围的插件文件，可后续补充规则”

### 6.6 测试面要扩展

本次改动至少影响以下测试面：

- CLI `scan`
- usecase `Scan`
- detector 状态判定
- 默认规则 definitions
- accessor 白名单
- snapshot collector 收集规则
- TUI 扫描结果显示

## 7. 建议的落地方向

建议将本次需求视为“同步规则统一化”工作，而不是单独的 `scan` 小改动。

推荐实现方向：

1. 抽出统一的“同步规则解析层”，作为 `scan`、`get`、`snapshot create` 的共同来源。
2. 默认规则与用户自定义规则在同一层合并，得到最终生效规则。
3. `scan` 负责展示：
   - 默认规则命中项
   - 自定义命中项
   - 未命中自定义项
   - 关键文件缺失原因
4. `get` 只允许访问最终生效规则中的路径。
5. `snapshot create` 只打包最终生效规则命中的文件。

## 8. 结论

本次需求表面上是 `scan` 命令增强，实质上是一次规则层收口工作。

如果只做 CLI 参数和展示层改动，将产生以下风险：

- `scan` 展示范围与 `snapshot create` 实际打包范围不一致
- `scan add` 后 `get` 无法访问新增路径
- CLI 与 TUI 状态展示不一致
- 文档继续沿用旧规则造成误导

因此建议后续实施以“统一规则源”为主线推进。
