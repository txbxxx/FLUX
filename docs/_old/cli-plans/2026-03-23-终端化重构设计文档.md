# AI Sync Manager CLI/TUI MVP 设计文档

## 文档信息

- 日期：2026-03-23
- 范围：CLI/TUI 第一阶段 MVP
- 当前状态：已确认先做本地闭环，不进入前端桌面界面增强
- 设计目标：先把终端入口做成真正可用的产品雏形，再决定是否继续扩展桌面 GUI

## 1. 设计结论

第一阶段采用：

> `CLI first + thin TUI`

也就是先把非交互命令行流程做完整，再在同一套应用层之上加一个轻量终端界面。

第一阶段只覆盖本地闭环：

- 扫描本机工具配置
- 创建本地快照
- 列出本地快照

明确不做：

- 远端仓库连接
- push / pull
- diff / apply / rollback
- 凭证和加密配置界面

## 2. 当前代码上下文

当前仓库已经具备足够的后端能力，不需要从零开始设计核心业务逻辑。

现有关键能力如下：

- `internal/service/tool/` 已实现 Codex / Claude 配置扫描
- `internal/service/snapshot/` 已实现本地快照创建、列表、删除、导出、应用
- `pkg/database/` 已实现 SQLite 初始化与迁移
- `app.go` 已将这些服务装配到 Wails 绑定层

因此，本阶段的核心不是“补业务”，而是：

1. 把服务初始化从 Wails 绑定中抽离
2. 增加可复用的应用层
3. 在其上实现 CLI 和 TUI 两种终端入口

## 3. 方案选择过程

讨论后的备选方案有三种。

### 方案 A：共享应用层 + 标准库 CLI + 菜单式 TUI

特点：

- 抽出共享 runtime 和 usecase
- CLI 使用 Go 标准库子命令模式
- TUI 先做菜单驱动、页面切换明确的终端界面

优点：

- 不引入额外依赖，推进速度快
- 架构边界清晰
- 后续可以平滑升级为更复杂的终端 UI
- 对当前 MVP 最符合 YAGNI

缺点：

- TUI 第一版不会是复杂的全屏富交互界面
- 视觉表达能力弱于成熟终端框架

### 方案 B：共享应用层 + 第三方 CLI/TUI 框架

特点：

- 一开始就引入成熟的 CLI 框架和 TUI 框架
- 结构更现代，交互更强

优点：

- 开发体验更好
- 后续扩展命令和交互状态更顺手

缺点：

- 第一阶段就要处理依赖、框架约束和状态管理
- 对当前“先跑通闭环”的目标不够克制

### 方案 C：先做 TUI，不做完整 CLI

特点：

- 用户只通过终端界面完成扫描、快照、列表

优点：

- 交互统一
- 更接近“可展示产品”

缺点：

- 脚本化能力差
- 难以调试和验证底层行为
- 不利于后续自动化测试

最终采用：

- 以方案 A 为主

## 4. 架构决策

### 4.1 总体结构

第一阶段新增三层：

1. `runtime`：统一初始化日志、数据库、检测器、快照服务
2. `usecase`：把扫描 / 创建快照 / 列表组合成终端可调用的应用层
3. `entrypoints`：CLI 和 TUI

结构目标如下：

- Wails、CLI、TUI 都不直接拼装底层服务
- 终端层只处理输入、输出、状态切换
- 业务编排统一收敛到应用层

### 4.2 建议目录

建议新增以下目录：

```text
cmd/
  ai-sync/
    main.go

internal/
  app/
    runtime/
      runtime.go
    usecase/
      local_workflow.go
      local_workflow_test.go
  cli/
    root.go
    scan_cmd.go
    snapshot_cmd.go
    tui_cmd.go
    output.go
    root_test.go
  tui/
    model.go
    handlers.go
    render.go
    model_test.go
```

现有 `app.go` 不删除，但会改为依赖 `runtime` 和 `usecase`。

## 5. 功能范围定义

### 5.1 CLI 能力

第一阶段 CLI 提供 4 个入口：

- `ai-sync scan`
- `ai-sync snapshot create`
- `ai-sync snapshot list`
- `ai-sync tui`

#### scan

职责：

- 扫描本机全局 Codex / Claude 配置
- 输出工具状态、配置路径、识别到的配置项数量

第一阶段不做：

- 项目级批量扫描
- 自定义扫描目录
- JSON 以外的复杂机器可读输出格式

#### snapshot create

职责：

- 用户指定工具集合与 message
- 创建一条本地快照记录
- 输出快照 ID、名称、文件数、大小、创建时间

第一阶段默认行为：

- 默认范围为全局配置
- 不接入 tags
- 不做交互式冲突处理

#### snapshot list

职责：

- 列出本地数据库已有快照
- 默认按创建时间倒序显示
- 支持基础分页参数

第一阶段不做：

- 详情页
- 文件级展开
- 按工具和标签筛选

### 5.2 TUI 能力

第一阶段 TUI 采用“菜单式工作台”，不是复杂全屏仪表盘。

页面结构如下：

#### 首页

显示：

- 产品名称
- 当前数据目录
- 三个主动作

#### 扫描页

显示：

- 检测到的工具
- 每个工具的状态
- 配置项数量
- 返回主菜单入口

#### 创建快照页

交互：

- 选择工具
- 输入 message
- 执行创建
- 展示结果

#### 快照列表页

显示：

- 最近快照列表
- 基础摘要信息
- 返回主菜单入口

## 6. 数据流设计

### 6.1 scan

```text
CLI/TUI 输入
  -> usecase.Scan()
  -> tool detector
  -> 格式化结果
  -> CLI/TUI 输出
```

### 6.2 create snapshot

```text
CLI/TUI 输入
  -> usecase.CreateSnapshot()
  -> snapshot service.CreateSnapshot()
  -> database
  -> 返回摘要
  -> CLI/TUI 输出
```

### 6.3 list snapshots

```text
CLI/TUI 输入
  -> usecase.ListSnapshots()
  -> snapshot service.ListSnapshots()
  -> database
  -> 返回摘要列表
  -> CLI/TUI 输出
```

## 7. 错误处理设计

第一阶段要重点处理以下错误：

- 数据库初始化失败
- 未检测到任何工具
- 创建快照时工具列表为空
- 创建快照时未找到任何配置文件
- 列表为空

设计原则：

- CLI 输出一行主错误 + 一行建议动作
- TUI 输出简短错误摘要，并保留返回主菜单
- 不暴露底层堆栈信息给普通用户

示例方向：

- “未检测到任何可同步工具，请先确认 `~/.codex` 或 `~/.claude` 是否存在”
- “创建快照失败：未找到任何配置文件”

## 8. 测试策略

第一阶段优先保证三类测试：

### 8.1 usecase 单元测试

目标：

- 用 stub 验证应用层编排逻辑
- 不依赖真实文件系统和真实数据库

覆盖：

- 扫描结果映射
- 创建快照参数校验
- 列表排序和分页参数透传

### 8.2 CLI 命令测试

目标：

- 验证命令参数解析
- 验证标准输出文本
- 验证错误分支

### 8.3 TUI 状态测试

目标：

- 验证菜单跳转
- 验证创建成功后刷新列表
- 验证错误态能返回首页

第一阶段不强求：

- 复杂终端渲染快照测试
- 真实交互录屏验证

## 9. 风险与约束

### 9.1 当前 Wails 绑定耦合

现状：

- `app.go` 既负责初始化，又负责暴露绑定接口

风险：

- 如果不先抽 runtime，CLI/TUI 会重复初始化逻辑

应对：

- 先完成 runtime 抽离

### 9.2 快照能力边界

现状：

- 快照服务已可创建和列表
- 第一阶段不需要依赖远端同步与冲突处理

判断：

- 当前范围可安全避开同步侧未完成能力

### 9.3 TUI 复杂度膨胀

风险：

- 如果第一版就追求复杂终端框架和全屏状态，会拖慢进度

应对：

- 先用菜单式 TUI 保证闭环

## 10. 任务拆分结论

建议按以下顺序实施：

1. 抽 runtime 和 usecase
2. 完成 CLI 三个核心命令
3. 完成菜单式 TUI
4. 统一错误输出与状态刷新
5. 补测试和文档

## 11. 验收标准

第一阶段完成时，应满足：

- 可以通过 CLI 执行扫描、创建快照、列出快照
- 可以通过 TUI 完成同样流程
- 创建快照后，列表立即可见
- 没有工具、没有快照、创建失败时，都有明确反馈
- 不修改已有核心服务的业务语义
