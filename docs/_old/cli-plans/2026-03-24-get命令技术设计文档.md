# AI Sync Manager `get` 命令技术设计文档

## 文档信息

- 日期：2026-03-24
- 范围：为终端版 AI Sync Manager 设计 `get <app> <path>` 与 `--edit` 编辑模式
- 目标：在不破坏当前 `runtime + usecase + Cobra + Bubble Tea` 结构的前提下，新增配置浏览与配置编辑能力

## 1. 设计结论

本次功能采用以下技术组合：

- `Cobra`
  负责新增 `get` 子命令与 `--edit` 标志位
- `runtime + usecase`
  负责复用当前应用层结构，避免 CLI 直接操作文件系统细节
- `internal/service/tool`
  负责路径解析、范围校验、文件/目录读取、文件写回
- `Bubble Tea`
  负责 `--edit` 时的全屏编辑会话
- `Bubbles/help` 与 `Bubbles/key`
  负责编辑器底部快捷键提示
- 自定义编辑缓冲区
  负责多行文本编辑、行号显示、滚动与保存前脏状态管理
- Go 标准库 `os` / `filepath`
  负责真实文件系统访问与路径标准化

设计核心原则：

1. `get` 普通模式仍是 CLI 命令，不强制进入 TUI
2. 只有 `--edit` 才进入全屏终端编辑器
3. 文件访问必须限定在工具允许的配置范围内
4. 不引入第二套 TUI 框架，保持当前 Bubble Tea 技术栈一致

## 2. 当前代码上下文

当前仓库已经具备实现 `get` 的主体条件：

- `cmd/ai-sync/main.go`
  已统一入口
- `internal/cli/cobra/`
  已承接全部 CLI 命令树
- `internal/app/usecase/local_workflow.go`
  已是 CLI/TUI 共用的应用层入口
- `internal/service/tool/paths.go`
  已经定义 `Codex` / `Claude` 哪些路径属于配置范围
- `internal/tui/`
  已经接入 `Bubble Tea`
- `go.mod`
  已包含 `bubbletea` 与 `bubbles`

这意味着本次不需要引入新的架构方向，只需要在当前结构里补一条新的读写链路。

## 3. 方案比较

### 方案 A：继续使用当前技术栈，补 `get` 命令和内置编辑器

技术：

- Cobra
- Bubble Tea
- Bubbles `help/key`
- 自定义编辑缓冲区

优点：

- 与当前架构一致
- 不引入第二套事件模型
- 复用现有 TUI 依赖
- 编辑器能力可控
- 未来容易继续扩展只读预览、保存提示、快捷键帮助

缺点：

- 需要自己封装一个轻量编辑器模型
- 需要处理保存、脏状态、退出确认

### 方案 B：目录浏览自己做，`--edit` 调外部编辑器

技术：

- Cobra
- `exec.Command`
- 外部 `vim` / `nvim` / `notepad`

优点：

- 实现成本最低
- 编辑能力强

缺点：

- 行为依赖用户环境
- Windows / macOS / Linux 一致性差
- 无法保证统一快捷键与视觉体验
- 不符合“产品内置能力”的方向

### 方案 C：引入 `tview` 或 `tcell` 作为独立编辑器框架

技术：

- `rivo/tview`
- `gdamore/tcell`

优点：

- 组件丰富
- 官方明确提供多行文本区域和布局能力

缺点：

- 当前项目已经采用 Bubble Tea
- 引入第二套 TUI 框架会让交互层风格和状态模型分裂
- 增加维护成本和心智负担

最终采用：

- 方案 A

## 4. 技术选型说明

### 4.1 Cobra

用途：

- 提供 `ai-sync get <app> <path>`
- 提供 `--edit` 标志位

原因：

- 当前 CLI 已经全部迁到 Cobra
- 新命令应继续使用同一命令树，不应回退到手写命令解析

### 4.2 Bubble Tea

用途：

- 在 `--edit` 时启动一个全屏终端编辑会话
- 接收按键事件
- 处理窗口尺寸变化
- 退出后恢复正常终端屏幕

原因：

- 当前仓库已经使用 Bubble Tea 作为终端交互基础
- 官方 README 明确说明 Bubble Tea 适合构建简单和复杂的终端应用，并支持全窗口应用
- 当前版本支持 `WithAltScreen()`，可让编辑器进入 alternate screen，避免污染普通命令输出

为什么不用别的框架：

- 如果再引入 `tview` / `tcell`，项目将出现两套 TUI 技术栈
- 对当前项目来说，编辑器只是一个局部能力，不值得为它切换整套 UI 框架

### 4.3 自定义编辑缓冲区

用途：

- 多行文本编辑
- 光标移动
- 行号显示
- 垂直滚动
- 保存前脏状态判断

原因：

- 当前项目已经有 Bubble Tea 主循环，不需要第二套编辑组件框架
- 当前实际实现需求只覆盖轻量文本编辑，不需要完整 IDE 级能力
- 可以完全按项目需要控制键位、状态栏和保存逻辑

为什么这样选：

- 避免新增额外编辑组件依赖
- 避免被第三方组件的额外依赖链阻塞
- 当前需求下实现面更可控

限制：

- 默认不是 `vim` 的完整模态编辑器
- 当前只实现“类 vim 的终端全屏编辑体验”

### 4.4 Bubbles `help` / `key`

用途：

- 在编辑器底部显示快捷键提示

原因：

- 当前 TUI 已经用过 `help/key`
- 可以统一展示例如：
  - `Ctrl+S 保存`
  - `Esc 退出`
  - `PgUp/PgDn 滚动`

### 4.5 Go 标准库 `os` + `filepath`

用途：

- 路径清理
- 文件存在性判断
- 文件/目录判断
- 列目录
- 读取文件
- 写回文件

原因：

- 当前需求只需要访问本机真实文件
- Go 标准库已经足够
- 可以跨平台处理 Windows 与 Unix 风格路径

## 5. 为什么不选 `tview`

调研结论：

- `tview` 官方 README 明确表示它提供富交互终端组件，包括多行文本编辑区、表格、布局和应用封装
- 这说明 `tview` 本身确实有能力做这类编辑器

但本项目不采用它，原因是：

1. 当前项目主 TUI 已经基于 Bubble Tea
2. `get --edit` 只是一个局部编辑场景，不值得引入第二套 TUI 框架
3. 两套框架并存会让维护方式、测试方式、事件模型都分裂

因此，`tview` 技术上可行，但架构上不合适。

## 6. 为什么不调用外部编辑器

不采用以下方案：

- `exec.Command("vim", filePath)`
- `exec.Command("nvim", filePath)`
- `exec.Command("notepad", filePath)`

原因：

1. 用户环境不一致，不能假设一定装了 `vim` 或 `nvim`
2. Windows 上体验与 Unix 不统一
3. 交互能力、快捷键、退出码处理会依赖外部程序
4. 不利于后续把“查看”和“编辑”做成一致的产品体验

因此，本次 `--edit` 设计为内置能力。

## 7. 为什么不使用额外路径安全库

理论上可以引入专门的安全路径拼接库，但本次不建议新增。

原因：

1. 当前需求只处理“已存在的真实路径”
2. 当前项目需要保持跨平台
3. 使用标准库 `filepath.Clean`、`filepath.Abs`、`filepath.Rel`、`filepath.EvalSymlinks` 就足以覆盖本次越界校验需求

因此，本次设计采用：

- 路径标准化
- 真实路径解析
- 根目录前缀校验
- 允许范围白名单校验

## 8. 目标架构

### 8.1 总体数据流

普通目录/文件读取：

```text
Cobra get command
  -> usecase.GetConfig()
  -> tool.ConfigAccessor.Resolve()
  -> tool.ConfigAccessor.ListDir() / ReadFile()
  -> CLI formatter
  -> stdout
```

编辑模式：

```text
Cobra get --edit
  -> usecase.GetConfig()
  -> tool.ConfigAccessor.ReadFile()
  -> editor.Run()
  -> tool.ConfigAccessor.WriteFile()
  -> 返回 CLI
```

### 8.2 新增模块建议

建议新增以下文件：

```text
internal/
  app/
    usecase/
      config_browser.go

  service/

      get_test.go

  tui/

```

说明：

- `config_browser.go`
  放应用层入参、返回值和错误翻译
- `accessor.go`
  放真实文件系统访问和路径安全校验
- `get.go`
  放 CLI 子命令
- `editor*.go`
  放 `--edit` 的 Bubble Tea 编辑器

## 9. 应用层设计

### 9.1 新接口

建议在 usecase 层新增两类接口：

1. 浏览接口
2. 保存接口

示意：

```go
type ConfigBrowser interface {
    GetConfig(ctx context.Context, input GetConfigInput) (*GetConfigResult, error)
    SaveConfig(ctx context.Context, input SaveConfigInput) error
}
```

### 9.2 输入结构

建议输入结构：

```go
type GetConfigInput struct {
    Tool string
    Path string
    Edit bool
}
```

### 9.3 返回结构

建议返回结果区分目录与文件：

```go
type GetConfigResult struct {
    Tool         string
    RelativePath string
    AbsolutePath string
    Kind         ConfigTargetKind
    Entries      []ConfigEntry
    Content      string
    Editable     bool
}
```

其中：

- `Kind=directory` 时使用 `Entries`
- `Kind=file` 时使用 `Content`

## 10. 服务层设计

### 10.1 `ConfigAccessor` 职责

`internal/service/tool/accessor.go` 应负责：

- 根据工具名找到工具根目录
- 将用户输入的相对路径标准化
- 判断目标是否位于允许同步的配置范围内
- 判断目标是文件还是目录
- 读取目录一层内容
- 读取文本文件内容
- 写回文件内容

### 10.2 路径解析策略

推荐流程：

1. 根据 `toolType` 取默认全局配置根目录
2. 对用户输入路径做 `filepath.Clean`
3. 将清理后的相对路径拼到工具根目录
4. 使用 `filepath.Abs` 和 `filepath.EvalSymlinks` 获取真实路径
5. 校验真实路径仍位于工具根目录下
6. 校验真实路径位于工具定义允许访问的配置前缀下

### 10.3 白名单策略

白名单数据直接复用：

- `GetCodexFileDefinitions()`
- `GetClaudeFileDefinitions()`

处理方式：

- 先把定义中的顶级可访问项转换成允许前缀
- 用户请求路径必须命中这些前缀之一

例如：

- `codex` 允许：
  - `config.toml`
  - `skills\...`
  - `rules\...`
  - `superpowers\...`
- `claude` 允许：
  - `settings.json`
  - `CLAUDE.md`
  - `commands\...`
  - `plugins\...`
  - `output-styles\...`

### 10.4 文件与目录判断

统一基于 `os.Stat()`：

- `IsDir=true` -> 目录
- `IsDir=false` -> 文件

不能靠文件名后缀猜测。

### 10.5 目录列出规则

目录模式只返回当前一层子项：

- 调 `os.ReadDir`
- 不递归
- 输出时目录排前、文件排后
- 同类按名称排序

### 10.6 文件读取规则

只读文本文件时：

- 使用 `os.ReadFile`
- 以原样字符串返回

如果检测到二进制内容：

- 返回用户可读错误
- 编辑模式直接拒绝

### 10.7 文件保存规则

保存时建议使用“先写临时文件再原子替换”的方式：

1. 在同目录创建临时文件
2. 写入完整新内容
3. `fsync` 或 close 成功后替换原文件

这样可以降低中途失败导致原文件损坏的风险。

## 11. CLI 设计

### 11.1 新命令

根命令新增：

```text
ai-sync get <app> <path>
```

参数：

- `--edit`

### 11.2 命令行为

当 `--edit=false`：

- 若目标是目录，打印目录列表
- 若目标是文件，打印文件正文

当 `--edit=true`：

- 若目标是目录，报错，不进入编辑器
- 若目标是文件，进入内置编辑器

### 11.3 错误输出

继续复用 `usecase.UserError`，保持当前中文错误风格一致。

## 12. 编辑器设计

### 12.1 进入方式

只在 `get --edit` 时启动，不影响现有 `ai-sync tui` 主流程。

### 12.2 运行方式

推荐为单独的 Bubble Tea Program，而不是塞进现有首页式 TUI 中。

原因：

- 它是一次性会话，不是主菜单子页面
- 退出后应直接回到普通命令行
- 可以独立使用 `tea.WithAltScreen()`

### 12.3 UI 组成

编辑器界面建议包含：

- 顶部标题栏：工具名 + 相对路径
- 中间编辑区：`textarea.Model`
- 底部状态栏：修改状态、保存状态、光标位置
- 底部帮助栏：快捷键提示

### 12.4 组件职责

- `textarea`
  负责文本编辑、行号、光标、滚动
- `help/key`
  负责快捷键提示
- `lipgloss`
  负责顶栏、状态栏、错误样式

### 12.5 快捷键设计

建议键位：

- `Ctrl+S`：保存
- `Esc`：退出
- `Ctrl+C`：强制退出
- `PgUp/PgDn`：翻页
- `Home/End`：行首行尾

可选增强：

- 若未保存直接退出，弹确认提示

### 12.6 脏状态

编辑器需要维护：

- 原始内容
- 当前内容
- 是否已修改

只有内容变化时才显示“已修改”。

### 12.7 窗口尺寸变化

编辑器应处理 `tea.WindowSizeMsg`：

- 重新计算 textarea 宽高
- 保持顶栏和底栏高度固定

## 14. 风险与约束

### 14.1 `textarea` 的 clipboard 依赖

调研过程中确认 `bubbles/textarea` 依赖 `github.com/atotto/clipboard`。

影响：

- 理论上它是一个很合适的编辑组件
- 但也意味着编辑器会带上额外依赖链
- 在当前环境里，这条依赖链会放大本地 `go.sum` / 模块缓存问题

结论：

- 本次最终实现不采用 `textarea`
- 继续保留 Bubble Tea + Bubbles `help/key`
- 多行编辑能力改为项目内自定义实现

### 14.2 不是完整 Vim

本次只能做到：

- 全屏
- 行号
- 可滚动
- 可编辑
- 快捷键保存/退出

不会实现完整 Vim 模态命令体系。

因此文档中应使用“类似 vim 的终端编辑体验”，而不是“内置 Vim”。

### 14.3 文件安全

若直接覆盖写文件，失败时可能损坏原文件。

应对：

- 使用临时文件写入再替换

## 15. 验收标准

技术上完成时应满足：

- `get` 已注册为 Cobra 子命令
- 目录访问只列一层
- 文件访问输出正文
- `--edit` 进入全屏编辑器
- 编辑器带行号
- 保存后修改真实文件
- 访问范围被严格限制在允许配置路径内
- 目录不能进入编辑模式
- 测试覆盖浏览、编辑和安全边界

## 16. 结论

本次最合适的技术路线不是引入新的编辑器框架，而是在当前架构内继续使用：

> `Cobra + usecase + tool accessor + Bubble Tea + Bubbles help/key + 自定义编辑缓冲区`

这样做的原因很明确：

- 与当前仓库技术栈一致
- 依赖最少
- 架构边界最清楚
- 能满足“目录浏览 + 文件查看 + 全屏编辑 + 直接改本机文件”的需求

后续如果功能继续扩大，例如：

- 多文件标签页
- 搜索替换
- 差异对比
- 树状浏览器

再考虑是否把编辑器从“轻量局部能力”演进为“独立子系统”。
